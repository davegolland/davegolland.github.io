SONIC: Segmented Optimized Nexus for Information Compression in Key-Value Caching

SONIC: Segmented Optimized Nexus for Information Compression in Key-Value Caching

Source: arXiv:cs.CL Published: 2026-01-29 Rank: #5 (score: 0.93, max_sim: 0.82) Reading time: ~0 min Tags: Uncategorized

Abstract

The linear growth of Key-Value (KV) cache remains a bottleneck for multi-turn LLM deployment. Existing KV cache compression methods often fail to account for the structural properties of multi-turn dialogues, relying on heuristic eviction that risks losing critical context. We propose \textbf{SONIC}, a learning-based framework that compresses historical segments into compact and semantically rich \textbf{Nexus} tokens. By integrating dynamic budget training, SONIC allows flexible adaptation to varying memory constraints without retraining. Experiments show that at compression ratios of 80% and 50%, SONIC consistently outperforms baselines such as H2O and StreamingLLM on four diverse multi-turn benchmarks. Specifically, on the widely used MTBench101 benchmark, SONIC achieves an average score improvement of 35.55% over state-of-the-art baselines, validating its effectiveness in sustaining coherent multi-turn dialogues. Furthermore, SONIC enhances deployment efficiency, accelerating the overall inference process by 50.1% compared to full-context generation.

Dynamic Budget Training (DBT)

Core Idea: Train the compression model with variable budget constraints during training, enabling flexible adaptation to different memory constraints at inference time without retraining.

Problem It Solves: Fixed compression ratios during training limit deployment flexibility. Different serving environments have different memory constraints; retraining for each constraint is prohibitively expensive.

How It Works:

  1. During training, randomly sample compression budget values from a target range (e.g., compress to 50%, 65%, 80% of original cache size)
  2. For each training example, enforce the sampled budget constraint when compressing segments (e.g., produce fewer Nexus tokens when budget is tighter)
  3. Train the compression encoder with this variability, learning to produce high-quality Nexus representations across the entire budget range
  4. At inference time, specify the desired compression budget; the pre-trained model adapts to that constraint without additional training

Where It’s Useful:

  • Multi-tenant LLM serving where different customers have different hardware/memory constraints
  • Mobile or edge deployment where available memory varies by device
  • Auto-scaling inference infrastructure where resource allocation changes dynamically
  • Research and experimentation requiring rapid evaluation across compression trade-offs

Limitations / Caveats:

  • Wider training budget range may reduce compression quality at any specific budget point (accuracy-flexibility trade-off)
  • Requires careful hyperparameter tuning to balance adaptability across budget values
  • Performance at untested budget points may degrade if the range is too large

Internal Use Cases:

  • For diverse model serving: train ranking models with variable KV cache budgets to serve different latency/memory tiers
  • For adaptive inference: adjust compression budget based on query complexity at inference time

Multi-Segment Context-Aware Compression

Core Idea: Compress different dialogue segments with awareness of their relative importance to current generation task, rather than uniform compression across all history.

Problem It Solves: Naive uniform compression treats all historical context equally, but recent segments and semantically relevant segments are more critical than distant or irrelevant ones. This risks losing crucial context for coherent response generation.

How It Works:

  1. Score each historical segment based on semantic relevance to the current query and recent context (e.g., token overlap, attention patterns, or learned relevance scoring)
  2. Allocate compression budget preferentially to lower-relevance segments, compressing them more aggressively
  3. Preserve higher-relevance segments with less aggressive compression, retaining finer-grained detail
  4. Generate Nexus tokens for each segment respecting its allocated budget, producing a mixed-compression KV cache

Where It’s Useful:

  • Long multi-turn dialogues where only certain previous exchanges are relevant to the current turn
  • Complex conversations with topic switches where earlier irrelevant topics can be compressed more
  • Query-dependent context compression where the importance of history shifts based on user intent

Limitations / Caveats:

  • Relevance scoring adds computational overhead during preprocessing
  • Incorrect relevance estimates can degrade coherence (over-compressing relevant context)
  • Requires careful calibration of relevance scoring metric to avoid harmful over-compression

Internal Use Cases:

  • For retrieval-augmented ranking: compress less-relevant search results more aggressively while preserving high-NDCG results
  • For conversational recommendation: deprioritize historical user interactions less relevant to current query intent
