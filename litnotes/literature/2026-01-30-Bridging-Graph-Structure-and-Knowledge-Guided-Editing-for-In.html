Bridging Graph Structure and Knowledge-Guided Editing for Interpretable Temporal Knowledge Graph Reasoning

Source: arXiv:cs.LG Published: 2026-01-29 Rank: #6 (score: 0.99, max_sim: 0.83) Reading time: ~20 min Tags: LLM, Graph, RAG

Abstract

Temporal knowledge graph reasoning (TKGR) aims to predict future events by inferring missing entities with dynamic knowledge structures. Existing LLM-based reasoning methods prioritize contextual over structural relations, struggling to extract relevant subgraphs from dynamic graphs. This limits structural information understanding, leading to unstructured, hallucination-prone inferences especially with temporal inconsistencies. To address this problem, we propose IGETR (Integration of Graph and Editing-enhanced Temporal Reasoning), a hybrid reasoning framework that combines the structured temporal modeling capabilities of Graph Neural Networks (GNNs) with the contextual understanding of LLMs. IGETR operates through a three-stage pipeline. The first stage aims to ground the reasoning process in the actual data by identifying structurally and temporally coherent candidate paths through a temporal GNN, ensuring that inference starts from reliable graph-based evidence. The second stage introduces LLM-guided path editing to address logical and semantic inconsistencies, leveraging external knowledge to refine and enhance the initial paths. The final stage focuses on integrating the refined reasoning paths to produce predictions that are both accurate and interpretable. Experiments on standard TKG benchmarks show that IGETR achieves state-of-the-art performance, outperforming strong baselines with relative improvements of up to 5.6% on Hits@1 and 8.1% on Hits@3 on the challenging ICEWS datasets. Additionally, we execute ablation studies and additional analyses confirm the effectiveness of each component.

Key Techniques

  • Temporal GNN-based path extraction
  • LLM-guided path editing
  • Hybrid graph-language reasoning

LLM-Guided Path Editing and Refinement

Core Idea: Use LLMs to refine graph-extracted candidate paths by detecting and correcting logical and semantic inconsistencies while leveraging external knowledge to enhance reasoning quality.

Problem It Solves: GNN-extracted paths may contain semantically illogical sequences (e.g., “person graduated university” → “person won Nobel prize” → “person attended high school”) or miss context crucial for prediction. Path editing bridges the gap between structured graph reasoning and semantic understanding.

How It Works:

  1. Input candidate path from GNN stage: [entity_1 —r_1⇒ entity_2 —r_2⇒ … —r_n⇒ entity_n] with context window of temporal facts
  2. Prompt LLM with path, query, and external knowledge (e.g., ontology, commonsense facts) to identify:
    • Temporal contradictions (e.g., event happens before necessary prerequisite)
    • Semantic inconsistencies (relations do not naturally follow each other)
    • Missing intermediate entities or relations needed for logical coherence
  3. LLM generates correction suggestions and rationale explaining why edits improve the path
  4. Apply LLM-suggested edits to produce refined path: remove contradictory segments, reorder relation sequences, or insert missing intermediate entities
  5. Validate edited path against KG structure to ensure consistency

How It Works (Algorithm Detail):

  • Input: Path P = [(e_1, r_1, e_2), (e_2, r_2, e_3), …, (e_n, r_n, e_{n+1})], query Q = (s, r_query, ?, τ_query)
  • LLM Call: Encode P and Q into natural language prompt with instruction to identify logical gaps and temporal issues
  • Output: Sequence of edit operations {insert, remove, reorder, substitute} applied to path with confidence scores
  • Refinement: Apply edits in order of confidence, re-validating graph consistency after each operation

Architecture:

flowchart LR
    A["Candidate Path<br/>from GNN"] --> B["LLM Inspection<br/>with External<br/>Knowledge"]
    B --> C["Identify<br/>Inconsistencies"]
    C --> D["Generate<br/>Edit Operations"]
    D --> E["Apply Refinements<br/>& Validate"]
    E --> F["Refined Path<br/>(logically coherent)"]

Where It’s Useful:

  • Correcting multi-hop reasoning paths in QA systems with knowledge graphs
  • Refining retrieved evidence passages before using them for LLM-based prediction
  • Improving reasoning chains in complex event understanding with temporal constraints
  • Semantic validation of entity linking chains in knowledge base construction

Limitations / Caveats:

  • LLM editing can introduce new hallucinations if external knowledge is incorrect or outdated
  • Editing decisions are non-transparent; difficult to trace which knowledge source drove a specific refinement
  • Computationally expensive (per-path LLM inference); scales poorly with large candidate sets
  • LLM may over-correct, removing valid structural information in pursuit of semantic coherence
  • Sensitive to prompt engineering; small wording changes can significantly alter editing behavior

Internal Use Cases:

  • Refining retrieved ranking features in learning-to-rank models before final scoring
  • Validating retrieved context passages before feeding to LLM ranker for interpretability
  • Correcting temporal inconsistencies in user behavior sequences during session-based recommendations

Temporal Reasoning Path Integration and Prediction

Core Idea: Synthesize multiple refined reasoning paths from the GNN + LLM pipeline into a unified, interpretable prediction by aggregating evidence and assessing path consistency.

Problem It Solves: Multiple valid paths may support different entity predictions with varying confidence; naive aggregation loses interpretability. Integration must balance path consensus against diversity of valid reasoning while maintaining traceability.

How It Works:

  1. Collect set of refined paths from stage 2: {P_1, P_2, …, P_K} each supporting potential prediction (s, r_query, e_candidate_i, τ_query)
  2. Score each path on: support strength (GNN score + LLM refinement confidence), semantic plausibility, temporal consistency, and evidence diversity
  3. Aggregate path scores using weighted ensemble: each candidate entity e_i receives score S(e_i) = Σ_j weight(P_j) · match(P_j, e_i) where match returns 1 if path P_j supports e_i
  4. Rank candidates by aggregated score and return top-1 prediction with supporting paths for interpretability
  5. Optionally, use path diversity as confidence calibration: predictions supported by diverse reasoning paths are more reliable

Scoring Detail:

  • Support Strength: Combined GNN structural score + LLM refinement confidence (1-5 scale) normalized to [0,1]
  • Temporal Consistency: Binary check if path respects τ_query; discounted if timeline ambiguous
  • Path Diversity Weight: Boost scores for predictions supported by structurally different paths (measured via path graph edit distance)

Architecture:

flowchart LR
    A["Refined Paths<br/>P_1, P_2, ..., P_K"] --> B["Score Each Path<br/>Structural + Semantic<br/>+ Temporal"]
    B --> C["Candidate Entity<br/>Evidence Aggregation"]
    C --> D["Rank Candidates<br/>by Aggregated<br/>Score"]
    D --> E["Return Prediction<br/>+ Supporting Paths"]

Where It’s Useful:

  • Final ranking stage in temporal event prediction with multiple reasoning chains
  • Interpretable entity linking by returning top paths that justify disambiguation
  • Confidence estimation in knowledge graph completion by path diversity metrics
  • Explainable recommendations by surfacing reasoning paths to users

Limitations / Caveats:

  • Aggregation weights are manually tuned; optimal weighting may be domain-specific
  • Path diversity metric (graph edit distance) is expensive to compute for large path sets
  • Predictions supported by few high-confidence paths may be brittle and domain-specific
  • Interpretability benefit degrades if paths themselves are complex; user-facing explanations require further summarization

Internal Use Cases:

  • Aggregating multiple ranking signals (structural, semantic, temporal) into final ranking score
  • Calibrating confidence scores for ranking predictions based on path diversity (more diverse supporting evidence = higher confidence)
  • Generating explainable ranking justifications by selecting and presenting top supporting paths to users
