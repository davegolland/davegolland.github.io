TAM-Eval: Evaluating LLMs for Automated Unit Test Maintenance

Source: HuggingFace Published: 2026-01-26 Rank: #9 (score: 0.98, max_sim: 0.83) Reading time: ~20 min Tags: LLM, RAG, Search Code: https://github.com/trndcenter/TAM-Eval

Abstract

While Large Language Models (LLMs) have shown promise in software engineering, their application to unit testing remains largely confined to isolated test generation or oracle prediction, neglecting the broader challenge of test suite maintenance. We introduce TAM-Eval (Test Automated Maintenance Evaluation), a framework and benchmark designed to evaluate model performance across three core test maintenance scenarios: creation, repair, and updating of test suites. Unlike prior work limited to function-level tasks, TAM-Eval operates at the test file level, while maintaining access to full repository context during isolated evaluation, better reflecting real-world maintenance workflows. Our benchmark comprises 1,539 automatically extracted and validated scenarios from Python, Java, and Go projects. TAM-Eval supports system-agnostic evaluation of both raw LLMs and agentic workflows, using a reference-free protocol based on test suite pass rate, code coverage, and mutation testing. Empirical results indicate that state-of-the-art LLMs have limited capabilities in realistic test maintenance processes and yield only marginal improvements in test effectiveness. We release TAM-Eval as an open-source framework to support future research in automated software testing. Our data and code are publicly available at https://github.com/trndcenter/TAM-Eval.

Reference-Free Evaluation Protocol

Core Idea: A metric system for test maintenance that uses observable test outcomes (pass/fail, coverage, mutation score) rather than oracle comparison, enabling evaluation without ground-truth test implementations.

Problem It Solves: Traditional test evaluation requires expert-written reference implementations or test oracles. Reference-free evaluation avoids this bottleneck while focusing on practical test effectiveness metrics that matter for production systems.

How It Works:

  1. Execute target test suite against codebase to determine pass rate: binary outcome of test_execution_result ∈ {pass, fail}
  2. Instrument code to measure code coverage: Δcoverage = coverage(modified_tests) - coverage(original_tests) across statements/branches
  3. Apply mutation testing framework to inject syntactic faults into source code
  4. Run modified test suite against mutants to calculate mutation score: mutation_score = (killed_mutants / total_mutants) × 100%
  5. Aggregate three metrics into composite evaluation without requiring reference test implementations

Where It’s Useful:

  • Evaluating test generation when oracle specification is unavailable
  • Benchmarking test repair without reference implementations
  • Measuring test quality improvements in continuous integration
  • Cross-project evaluation where reference tests differ in style/completeness

Limitations / Caveats:

  • Pass rate is coarse-grained (binary)—doesn’t capture test assertion quality or semantic correctness
  • Coverage metric biased toward code size; low coverage may reflect unmaintained code rather than test weakness
  • Mutation testing expensive (scales O(|mutants| × |test_executions|)); limits scope on large test suites
  • Cannot detect tests that check wrong properties (passing mutants despite incorrect behavior)
  • Assumes mutation testing framework availability for target language

Internal Use Cases:

  • Evaluating LLM test generation for recommendation systems without hand-written test oracles
  • Measuring test suite effectiveness in ranking model evaluation pipelines
  • Coverage-aware training signals for fine-tuning test generation models
