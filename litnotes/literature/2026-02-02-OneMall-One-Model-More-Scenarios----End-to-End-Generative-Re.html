OneMall: One Model, More Scenarios -- End-to-End Generative Recommender Family at Kuaishou E-Commerce

OneMall: One Model, More Scenarios — End-to-End Generative Recommender Family at Kuaishou E-Commerce

Source: arXiv:cs.IR Published: 2026-01-29 Rank: #4 (score: 0.96, max_sim: 0.84) Reading time: ~0 min Tags: Uncategorized Code: https://arxiv.org/pdf/2601.21770v1

Abstract

In the wave of generative recommendation, we present OneMall, an end-to-end generative recommendation framework tailored for e-commerce services at Kuaishou. Our OneMall systematically unifies the e-commerce’s multiple item distribution scenarios, such as Product-card, short-video and live-streaming. Specifically, it comprises three key components, aligning the entire model training pipeline to the LLM’s pre-training/post-training: (1) E-commerce Semantic Tokenizer: we provide a tokenizer solution that captures both real-world semantics and business-specific item relations across different scenarios; (2) Transformer-based Architecture: we largely utilize Transformer as our model backbone, e.g., employing Query-Former for long sequence compression, Cross-Attention for multi-behavior sequence fusion, and Sparse MoE for scalable auto-regressive generation; (3) Reinforcement Learning Pipeline: we further connect retrieval and ranking models via RL, enabling the ranking model to serve as a reward signal for end-to-end policy retrieval model optimization. Extensive experiments demonstrate that OneMall achieves consistent improvements across all e-commerce scenarios: +13.01% GMV in product-card, +15.32% Orders in Short-Video, and +2.78% Orders in Live-Streaming. OneMall has been deployed, serving over 400 million daily active users at Kuaishou.

Key Techniques

  • E-commerce Semantic Tokenizer
  • Transformer-based Architecture with Query-Former and Sparse MoE
  • Reinforcement Learning for retrieval-ranking connection
  • end-to-end generative recommendation

Query-Former: Long Sequence Compression

Core Idea: A Transformer-based module that compresses long user behavior sequences into a fixed-size compact representation while preserving salient patterns for downstream recommendation.

Problem It Solves: User behavior sequences in e-commerce can be very long (hundreds of interactions), creating computational bottlenecks for auto-regressive generation and attention mechanisms. Standard attention has quadratic complexity in sequence length, making long sequences impractical.

How It Works:

  1. Input: Long user behavior sequence S = [t₁, t₂, …, tₙ] where n can be several hundred tokens
  2. Apply learnable query tokens (compact tokens learned during training) that attend over the full sequence via cross-attention
  3. Each query token computes: Query_i = Attention(q_i, S) where q_i is a learnable query vector attending to all behavior tokens
  4. Output: Compressed representation C = [Query₁, Query₂, …, Query_k] where k << n, retaining critical interaction patterns

Architecture:

flowchart LR
    A["Long Sequence S<br/>n tokens"] --> B["Cross-Attention<br/>Learnable Queries"]
    B --> C["Compressed Rep C<br/>k tokens k<<n"]
    C --> D["Downstream<br/>Transformer"]

Where It’s Useful:

  • Reducing computational cost in sequence-to-sequence recommendation models
  • Enabling long-context understanding without quadratic attention cost
  • Pre-processing step for generative ranking models with memory constraints

Limitations / Caveats:

  • Determines optimal number of queries k (information bottleneck trade-off)
  • Learnable queries may not capture all important interaction patterns
  • Compression is lossy and cannot recover original sequence details

Internal Use Cases:

  • Compressing user click histories before passing to ranking LLMs
  • Handling cold-start sequences with limited compression capability

Cross-Attention Multi-Behavior Sequence Fusion

Core Idea: A cross-attention mechanism that integrates multiple heterogeneous user behavior sequences (clicks, purchases, watches, shares) into a unified representation for ranking decisions.

Problem It Solves: Users exhibit diverse interaction types in e-commerce (clicking, purchasing, watching, adding to cart, sharing), each with different semantic importance. Treating these behaviors separately loses correlative information; naive concatenation treats them equally. This method learns relative importance across behavior types.

How It Works:

  1. Input: Multiple behavior sequences B₁ (clicks), B₂ (purchases), B₃ (watches), etc., each compressed by Query-Former
  2. Designate one sequence as primary (e.g., purchases) and others as context sequences
  3. Apply cross-attention from primary sequence to each context sequence: Fusion_i = Attention(B_primary, B_i)
  4. Learn behavior-specific importance weights: w_i learned during training determine relative contribution of each behavior type
  5. Output: Fused representation F = Σᵢ w_i · Fusion_i

Architecture:

flowchart LR
    A["Behavior 1<br/>Clicks"] --> C["Cross-Attention<br/>Fusion Module"]
    B["Behavior 2<br/>Purchases"] --> C
    D["Behavior 3<br/>Watches"] --> C
    C --> E["Unified<br/>Representation"]

Where It’s Useful:

  • Multi-behavior recommendation systems where users leave different interaction signals
  • Learning relative importance of clicks vs purchases vs engagement signals
  • Fusion of explicit and implicit feedback in ranking models

Limitations / Caveats:

  • Requires clear semantic distinction between behavior types; ambiguous behaviors are hard to fuse
  • Learned importance weights may not transfer across different item catalogs or user populations
  • Assumes behavior types are complementary; some behaviors may be redundant

Internal Use Cases:

  • Fusing click history with conversion signals for e-commerce ranking
  • Combining explicit ratings with engagement duration for video recommendation

Sparse Mixture-of-Experts (Sparse MoE) for Auto-Regressive Generation

Core Idea: A scalable architecture where only a sparse subset of expert networks are activated for each prediction, enabling efficient auto-regressive token generation at scale.

Problem It Solves: Generative recommendation requires auto-regressive decoding (predicting next item token sequentially), which is computationally expensive at scale. Dense Transformer layers become prohibitive for generating long item sequences. Standard MoE improves capacity but requires careful load balancing.

How It Works:

  1. Input: Compressed user representation h (from Query-Former + Cross-Attention)
  2. Route input through gating network: g(h) = softmax(Wh), producing sparse selection distribution over k experts
  3. Select top-s experts (e.g., s=2 out of k=16 experts) based on gating scores
  4. Forward through selected experts: output = Σᵢ∈top-s g_i(h) · Expert_i(h)
  5. Generate next item token: token_{t+1} = argmax(softmax(output · W_vocab))
  6. Repeat for t = 1 to sequence_length to generate full recommendation sequence

Architecture:

flowchart LR
    A["User Rep h"] --> B["Gating Network<br/>g(h)"]
    B --> C["Select Top-s<br/>Experts"]
    C --> D["Expert 1<br/>Expert 2"]
    D --> E["Merge Outputs"]
    E --> F["Vocab Projection<br/>& Softmax"]
    F --> G["Next Token<br/>token_t+1"]

Where It’s Useful:

  • Scaling generative models to large catalogs (100K+ items)
  • Auto-regressive recommendation where you predict sequence of items
  • Production systems requiring efficient inference (only s/k of computation)

Limitations / Caveats:

  • Load balancing across experts is non-trivial; some experts may be unused while others overload
  • Sparse routing introduces gradient flow issues during training (vanishing gradients to unused experts)
  • Top-k selection is non-differentiable; requires approximations (straight-through estimator or relaxed routing)
  • Optimal expert count and selection size k, s require tuning per problem

Internal Use Cases:

  • Scaling ranking models to millions of candidate items
  • Efficient auto-regressive decoding in LLM-based recommendation systems

End-to-End Reinforcement Learning Pipeline for Retrieval-Ranking Optimization

Core Idea: An RL framework that treats the ranking model as a learned reward signal and optimizes the retrieval model end-to-end to maximize business metrics (GMV, Orders), enabling joint optimization of both stages.

Problem It Solves: Traditional retrieval and ranking are optimized separately (retrieval maximizes recall, ranking maximizes relevance), leading to suboptimal end-to-end performance. The retrieval model cannot see the ranking model’s preferences, causing inefficient candidate pool selection. RL bridges this gap by optimizing retrieval for final business outcomes.

How It Works:

  1. Retrieval Policy π_R: Maps user context u to candidate set C = π_R(u) (e.g., top-k items from embedding search)
  2. Ranking Reward Model R_rank: Scores candidates via trained ranking model, produces ranking R_rank(C, u) = [s₁, s₂, …, sₖ]
  3. Define reward signal from ranking scores and business metrics: r(C, u) = λ₁ · business_metric(C, u) + λ₂ · ranking_score(C, u)
  4. Train retrieval policy via policy gradient: ∇J = E[∇ log π_R(C|u) · r(C, u)], updating retrieval to select candidates that ranking model scores highly AND maximize business outcomes
  5. Ranking model provides learning signal (gradient) back to retrieval, enabling joint optimization

Architecture:

flowchart LR
    A["User Context u"] --> B["Retrieval Policy<br/>π_R"]
    B --> C["Candidate Set C"]
    C --> D["Ranking Model<br/>R_rank"]
    D --> E["Ranking Scores"]
    E --> F["Reward Signal r"]
    F --> G["Policy Gradient<br/>∇J"]
    G --> B

Where It’s Useful:

  • Two-stage recommendation systems (retrieval + ranking)
  • Optimizing for business KPIs (GMV, conversion, orders) rather than just relevance
  • Systems where top-k retrieval decisions significantly impact final recommendations
  • Online A/B testing where business metrics can be measured

Limitations / Caveats:

  • High variance in policy gradient estimates; requires careful variance reduction (baseline, advantage normalization)
  • Ranking model must be differentiable or use score-based reward (non-differentiable ranking)
  • Requires significant online interaction data to estimate reliable gradients
  • Sparse reward signal (business metrics measured at end of episode) makes learning slow
  • Changes to ranking model require retraining retrieval policy; tight coupling creates instability

Internal Use Cases:

  • Optimizing two-stage ranking systems for revenue metrics
  • Joint training of dense retrieval and learned-to-rank models
  • Closing the loop between information retrieval and downstream business outcomes
