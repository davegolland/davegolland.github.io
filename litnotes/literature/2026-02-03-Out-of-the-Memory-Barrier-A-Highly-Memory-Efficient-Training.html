Out of the Memory Barrier: A Highly Memory Efficient Training System for LLMs with Million-Token Contexts

Out of the Memory Barrier: A Highly Memory Efficient Training System for LLMs with Million-Token Contexts

Source: arXiv:cs.CL Published: 2026-02-02 Rank: #1 (score: 0.93, max_sim: 0.86) Reading time: ~0 min Tags: Uncategorized Code: https://arxiv.org/pdf/2602.02108v1

Abstract

Training Large Language Models (LLMs) on long contexts is severely constrained by prohibitive GPU memory overhead, not training time. The primary culprits are the activations, whose memory footprints scale linearly with sequence length. We introduce OOMB, a highly memory-efficient training system that directly confronts this barrier. Our approach employs a chunk-recurrent training framework with on-the-fly activation recomputation, which maintains a constant activation memory footprint (O(1)) and shifts the primary bottleneck to the growing KV cache. To manage the KV cache, OOMB integrates a suite of synergistic optimizations: a paged memory manager for both the KV cache and its gradients to eliminate fragmentation, asynchronous CPU offloading to hide data transfer latency, and page-level sparse attention to reduce both computational complexity and communication overhead. The synergy of these techniques yields exceptional efficiency. Our empirical results show that for every additional 10K tokens of context, the end-to-end training memory overhead increases by a mere 10MB for Qwen2.5-7B. This allows training Qwen2.5-7B with a 4M-token context on a single H200 GPU, a feat that would otherwise require a large cluster using context parallelism. This work represents a substantial advance in resource efficiency for long-context LLM training. The source code is available at https://github.com/wenhaoli-xmu/OOMB.

Paged Memory Manager for KV Cache and Gradients

Core Idea: Organize KV cache and gradient memory into fixed-size pages with dynamic allocation, eliminating fragmentation and enabling efficient offloading to CPU memory.

Problem It Solves: KV cache grows linearly with sequence length and becomes the dominant memory bottleneck after activations are eliminated. Contiguous memory allocation leads to fragmentation and external memory fragmentation, preventing efficient use of available GPU memory and CPU offloading.

How It Works:

  1. Allocate KV cache in fixed-size pages (similar to virtual memory paging in operating systems)
  2. Maintain a page allocation table tracking which pages are in use and their memory locations
  3. For sequences longer than GPU memory allows, allocate overflow pages in CPU RAM
  4. Track page locations (GPU vs CPU) during forward/backward passes
  5. Pages are deallocated and freed for reuse as tokens are processed and context windows shift
  6. Gradient pages follow the same paging mechanism, allowing gradients to also spill to CPU if needed

Architecture:

flowchart TD
    A["KV Cache Request<br/>Size = seq_len * d_model"] --> B{"Page<br/>Available?"}
    B -->|Yes| C["Allocate from GPU<br/>Page Pool"]
    B -->|No| D{"GPU Memory<br/>Full?"}
    D -->|No| C
    D -->|Yes| E["Allocate from CPU<br/>Page Pool"]
    C --> F["Track Page Location<br/>in Allocation Table"]
    E --> F
    F --> G["Return Page Handle<br/>to Model"]

Where It’s Useful:

  • Training on sequences exceeding single GPU memory capacity
  • Systems with heterogeneous memory (GPU + CPU RAM available)
  • Long-context inference scenarios where KV cache grows unbounded
  • Batch training where KV cache varies significantly across examples

Limitations / Caveats:

  • Page-based allocation introduces indexing overhead during access
  • CPU offloading introduces PCIe bandwidth bottleneck for data transfer
  • Page size selection affects fragmentation vs. overhead trade-off
  • Gradient computation may need to be stalled if gradient pages are on CPU while weights are on GPU

Internal Use Cases:

  • Managing cache for long user sequences in recommendation systems
  • Training ranking models with variable-length context windows without manual sequence padding

Asynchronous CPU Offloading with Latency Hiding

Core Idea: Proactively transfer KV cache pages to CPU memory asynchronously during computation phases, overlapping data movement latency with computation to maintain GPU utilization.

Problem It Solves: When KV cache overflows to CPU due to GPU memory constraints, synchronous transfers block computation, creating idle GPU time. Naive offloading strategies result in significant performance degradation as GPUs wait for data transfers.

How It Works:

  1. Predict which KV cache pages will not be needed in the next N computation steps
  2. Initiate asynchronous PCIe transfers to move these pages from GPU to CPU VRAM during forward/backward computation
  3. Overlap transfer latency with GPU computation on independent data
  4. When a page on CPU is needed again, prefetch it back to GPU asynchronously before it’s actually required
  5. Use a scheduling algorithm to coordinate transfers such that data arrives just-in-time without stalling computation
  6. Maintain a pool of available GPU memory pages for rapid re-loading of frequently-accessed offloaded pages

Architecture:

flowchart LR
    A["Computation Queue<br/>Steps t:t+N"] --> B["Identify Unused<br/>Pages in t:t+N"]
    B --> C["Async Transfer:<br/>GPU → CPU"]
    D["GPU Computation<br/>Progress"] --> E{"Transfer<br/>Complete?"}
    E -->|Yes| F["Page in CPU Memory"]
    E -->|No| D
    G["Future Step t+N<br/>Needs Page"] --> H["Async Transfer:<br/>CPU → GPU"]
    H --> I["Page Ready<br/>for Computation"]

Where It’s Useful:

  • GPU systems with access to large CPU RAM pools
  • Scenarios where context length grows dynamically during training
  • Batch inference with variable sequence lengths per example
  • Multi-GPU training where some GPUs have less memory than others

Limitations / Caveats:

  • Requires accurate prediction of future page access patterns (mispredictions cause stalls)
  • PCIe bandwidth limits the rate of offloading (typically 16-32 GB/s, much slower than GPU memory)
  • Asynchronous operations add complexity and potential synchronization bugs
  • Effective only if computation can be overlapped; small models may not have enough work to hide latency
  • Overhead grows with distance between GPU and CPU memory (NUMA effects on larger systems)

Internal Use Cases:

  • Training recommendation models with long user history where most recent interactions compute while older ones offload
  • Multi-turn conversational models where early context is offloaded during processing of recent turns

Page-Level Sparse Attention

Core Idea: Apply sparse attention patterns at the granularity of pages (blocks of tokens) rather than individual tokens, reducing both computational complexity and KV cache access overhead.

Problem It Solves: Full attention over millions of tokens requires O(n²) computations and memory access, which remains prohibitively expensive even with KV cache on CPU. Sparse attention reduces complexity, but token-level sparsity introduces irregular memory access patterns and poor hardware utilization.

How It Works:

  1. Organize tokens into fixed-size pages (e.g., 64 or 128 tokens per page)
  2. Define sparse attention pattern at page level: which query pages can attend to which key-value pages
  3. For each query page, compute attention only over a sparse subset of KV cache pages (e.g., local pages + summary pages)
  4. Within selected pages, optionally apply dense attention (all tokens in query page attend to all tokens in selected KV pages)
  5. This two-level hierarchy (sparse at page level, dense within pages) exploits both sequential locality and hardware efficiency
  6. Pattern can be block-diagonal, strided, local-window-plus-summary, or custom based on task requirements

Architecture:

flowchart LR
    A["KV Cache Pages<br/>P1, P2, P3...Pn"] --> B["Sparse Pattern<br/>at Page Level"]
    C["Query Pages<br/>Q1, Q2, Q3"] --> D["Page Attention<br/>Matrix"]
    B --> D
    D --> E["Select KV Pages<br/>for Each Q Page"]
    E --> F["Dense Attention<br/>Within Selected Pages"]
    F --> G["Output<br/>Attention Scores"]

Where It’s Useful:

  • Training extremely long-context models (1M-10M tokens) where full attention is infeasible
  • Retrieval-augmented generation where only relevant context pages matter
  • Time-series or document analysis where locality is important
  • Any attention-based model where full O(n²) attention is prohibitive

Limitations / Caveats:

  • Reduces model expressiveness compared to full attention (some token pairs never interact)
  • Sparse pattern design is task-dependent; wrong pattern choice hurts performance
  • Page size selection affects the sparsity-computation trade-off
  • May require careful tuning to avoid ignoring important long-range dependencies
  • Implementation complexity increases compared to standard dense attention
  • Sparse patterns with irregular access can still have poor cache locality if not carefully designed

Internal Use Cases:

  • Long-context ranking where candidate relevance is determined by recent interactions (local pages) and historical patterns (summary pages)
  • Multi-document retrieval ranking where attention only focuses on relevant documents (pages) rather than all tokens

Extending Long-Context Training Beyond Efficient Attention

While the matched notes on “Efficient Attention Mechanisms for Large Language Model: A Survey” focus on reducing the computational complexity of attention through linear approximations and architectural innovations (like Mamba and RetNet), OOMB addresses a complementary but distinct bottleneck: the memory overhead of training with long contexts. The survey emphasizes how linear attention variants achieve linear time/space scaling for inference, but OOMB’s insight is that even with efficient attention, the activation memory during training scales linearly with sequence length. By introducing O(1) activation memory through chunk-recurrent training with on-the-fly recomputation, OOMB sidesteps the assumption that longer context windows necessarily require more activation storage—a perspective that complements rather than contradicts efficient attention research. The two approaches are synergistic: efficient attention mechanisms can reduce per-token computation, while OOMB reduces per-token activation memory, together enabling practical long-context training.

Bridging Pretraining Context and Training Feasibility

The “Lengthy Context Retrieval via LLM Pretraining” note describes leveraging LLMs pre-trained on large context windows to enable long-context inference without expensive fine-tuning. OOMB directly addresses the inverse problem: how to make long-context pretraining itself feasible. The note assumes that large-context pretraining is already possible (e.g., 8k+ token windows), but OOMB’s contribution is making this assumption practical for resource-constrained settings—enabling 4M-token context pretraining on a single H200 GPU. This reframes the relationship: OOMB provides the infrastructure that makes the pretraining stage of the pipeline in the matched note economically viable, particularly for organizations without massive compute clusters. The paged KV cache memory manager and asynchronous CPU offloading are specifically engineered solutions to the training-time constraints that Lengthy Context Retrieval takes as solved.

Addressing Context Rot Through Training Infrastructure

The “Long Context Degradation in LLMs” note warns that current benchmarks like Needle in a Haystack obscure real performance degradation on semantically complex tasks with distractors—a phenomenon called context rot. OOMB’s significance here is methodological: by making long-context training dramatically more accessible (eliminating the cluster requirement), it enables broader empirical investigation of context degradation across model scales and training regimes. Previously, studying context rot at million-token scales required institutional-scale compute; OOMB democratizes this investigation. Furthermore, the paper’s chunk-recurrent framework with page-level sparse attention may naturally induce different attention patterns than full-context training, potentially offering a fresh angle on understanding why context degradation occurs and whether certain training approaches mitigate it better than others.
