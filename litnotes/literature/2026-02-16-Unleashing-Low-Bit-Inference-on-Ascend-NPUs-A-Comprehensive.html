Abstract

As LLMs scale, low-bit floating-point formats like MXFP and NVFP4 offer new opportunities for precision and efficiency. In this work, we evaluate HiFloat (HiF8 and HiF4), a family of formats tailored for Ascend NPUs. Through rigorous comparison across weight-activation and KV-cache tasks, we provide three key insights: (1) INT8 suits narrow-range data, while floating-point formats excel with high-variance data; (2) in 4-bit regimes, HiF4’s hierarchical scaling prevents the accuracy collapse seen in integer formats; and (3) HiFloat is fully compatible with state-of-the-art post-training quantization frameworks. Overall, HiFloat provides a solution for high-efficiency LLM inference on NPUs.

HiFloat Low-Bit Quantization Formats (HiF8/HiF4)

Core Idea: A family of hierarchical floating-point formats (HiF8 and HiF4) designed for Ascend NPUs that use block-level scaling factors to preserve precision in high-variance data during LLM inference quantization.

Problem It Solves: Integer quantization (INT4/INT8) suffers from accuracy collapse on high-variance activations and KV-cache data; existing floating-point formats like MXFP and NVFP4 are not optimized for Ascend NPU hardware characteristics.

How It Works:

  1. Block-based scaling: Divide weight/activation tensors into fixed-size blocks (e.g., 128 elements), compute a shared scaling factor per block
  2. Hierarchical precision: HiF8 uses 8-bit floating-point representation within blocks; HiF4 uses 4-bit with more aggressive quantization while maintaining the block-scale hierarchy
  3. Format-specific encoding: Apply exponent/mantissa bit allocation optimized for NPU compute units (different from MXFP’s tile-based or NVFP4’s GPU-centric design)
  4. Runtime dequantization: NPU hardware natively supports HiFloat formats, performing scale-and-dequantize operations inline during matrix operations

Architecture:

flowchart LR
    A[FP16 Weights/Activations] --> B[Block Partitioning]
    B --> C[Per-Block Scale Computation]
    C --> D[HiF8/HiF4 Encoding]
    D --> E[NPU MatMul Units]
    E --> F[Scale-Aware Accumulation]
    F --> G[FP16 Output]

Where It’s Useful:

  • LLM weight-only quantization (W4A16, W8A8 configurations)
  • KV-cache compression for long-context inference
  • Activation quantization in post-training quantization (PTQ) frameworks
  • Deployment on Ascend 910B/C NPUs where native HiFloat ops provide 2-4× throughput gains

Limitations / Caveats:

  • Hardware-specific: requires Ascend NPU support, not portable to NVIDIA/AMD GPUs without emulation
  • Block size sensitivity: optimal block size (32-128) varies by layer type; too small increases overhead, too large loses precision
  • 4-bit regime still shows 1-3% accuracy degradation vs FP16 on challenging benchmarks (MMLU, GSM8K), though better than INT4

Internal Use Cases:

  • LLM serving optimization: Deploy quantized Llama/Mistral models on Ascend infrastructure with 2× memory reduction (HiF8) or 4× (HiF4) while maintaining <1% accuracy loss
  • KV-cache management: Compress attention cache in long-context retrieval tasks (RAG systems) where high variance in key/value distributions makes INT8 unsuitable
  • Hybrid quantization: Combine HiF4 for weights (static, calibratable) with HiF8 for activations (dynamic, high-variance) in W4A8 mixed-precision schemes
