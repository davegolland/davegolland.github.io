ParisKV: Fast and Drift-Robust KV-Cache Retrieval for Long-Context LLMs

Source: arXiv:cs.CL Published: 2026-02-07 Rank: #6 (score: 0.93, max_sim: 0.86) Reading time: ~0 min Tags: Uncategorized

Abstract

KV-cache retrieval is essential for long-context LLM inference, yet existing methods struggle with distribution drift and high latency at scale. We introduce ParisKV, a drift-robust, GPU-native KV-cache retrieval framework based on collision-based candidate selection, followed by a quantized inner-product reranking estimator. For million-token contexts, ParisKV supports CPU-offloaded KV caches via Unified Virtual Addressing (UVA), enabling on-demand top- fetching with minimal overhead. ParisKV matches or outperforms full attention quality on long-input and long-generation benchmarks. It achieves state-of-the-art long-context decoding efficiency: it matches or exceeds full attention speed even at batch size 1 for long contexts, delivers up to 2.8 higher throughput within full attention’s runnable range, and scales to million-token contexts where full attention runs out of memory. At million-token scale, ParisKV reduces decode latency by 17 and 44 compared to MagicPIG and PQCache, respectively, two state-of-the-art KV-cache Top- retrieval baselines.

Key Techniques

  • Collision-based candidate selection
  • Quantized inner-product reranking
  • Unified Virtual Addressing (UVA) for CPU-offloaded KV caches

ParisKV (Partitioned Approximate Retrieval for Inference Speed)

Core Idea: A drift-robust, GPU-native KV-cache retrieval framework that uses collision-based candidate selection followed by quantized inner-product reranking to enable efficient top-k attention on long contexts, with CPU-offloading support via Unified Virtual Addressing for million-token scale.

Problem It Solves: Existing KV-cache retrieval methods for long-context LLMs suffer from distribution drift between prefill and decode phases, high retrieval latency at scale, and inability to handle million-token contexts due to memory constraints.

How It Works:

  1. Collision-based candidate selection: Hash query and key vectors into buckets; candidates are selected from colliding buckets, naturally handling distribution drift without retraining
  2. Quantized inner-product reranking: Compute approximate attention scores on candidates using low-precision quantized inner products to identify top-k most relevant KV pairs
  3. UVA-enabled CPU offloading: For million-token contexts, store KV cache on CPU and fetch only top-k entries on-demand via Unified Virtual Addressing with minimal overhead
  4. Sparse attention computation: Run full-precision attention only on retrieved top-k KV pairs, dramatically reducing memory and compute vs. dense attention

Where It’s Useful:

  • Long-document understanding and retrieval (RAG over 100k+ token contexts)
  • Multi-turn conversations with extensive history
  • Code generation with large codebases in context
  • Million-token scale inference where full attention is memory-infeasible
  • Low-latency decoding for real-time applications (matches full attention speed at BS=1)

Limitations / Caveats:

  • Requires careful tuning of hash bucket count and quantization precision to balance quality vs. speed
  • Collision-based selection may miss relevant tokens in adversarial distributions (though empirically robust)
  • CPU-offloading variant requires UVA-capable hardware (modern NVIDIA GPUs)
  • Quality-speed tradeoff: top-k selection inherently approximates full attention, though paper shows minimal degradation

Internal Use Cases:

  • Long-context ranking: Apply to reranking stages where queries need to attend over thousands of candidate documents
  • RAG systems: Enable efficient attention over massive retrieved context windows without memory blowup
  • LLM serving infrastructure: Integrate as a drop-in replacement for dense attention in decode phase to boost throughput 2-3x
  • Multi-document QA: Handle user queries over entire knowledge bases (millions of tokens) with sub-second latency
  • Hybrid with H2O/streaming attention: Combine ParisKV’s retrieval with heavy-hitter eviction for dynamic cache management
