LogitsCoder: Towards Efficient Chain-of-Thought Path Search via Logits Preference Decoding for Code Generation

Source: arXiv:cs.CL Published: 2026-02-15 Rank: #1 (score: 1.00, max_sim: 0.82) Reading time: ~20 min Tags: Search, RAG

Abstract

Code generation remains a challenging task that requires precise and structured reasoning. Existing Test Time Scaling (TTS) methods, including structured tree search, have made progress in exploring reasoning paths but still face two major challenges: (1) underthinking, where reasoning chains tend to be shallow and fail to capture the full complexity of problems; and (2) overthinking, where overly verbose reasoning leads to inefficiency and increased computational costs. To address these issues, we propose LogitsCoder, a novel framework that enhances chain-of-thought reasoning through lightweight, logit-level control mechanisms for code generation. LogitsCoder iteratively generates and refines reasoning steps by first steering token selection toward statistically preferred patterns via Logits Preference Decoding, then selecting and aggregating diverse reasoning paths using Logits Rank Based Path Selection and Thoughts Aggregation. This results in coherent and effective reasoning chains that balance depth and efficiency. Extensive experiments demonstrate that LogitsCoder produces more efficient and higher-quality reasoning chains, leading to superior code generation performance compared to baseline methods.

LogitsCoder

Core Idea: A test-time scaling framework that controls chain-of-thought reasoning at the logit level—biasing token selection toward statistically preferred patterns, then ranking and aggregating diverse reasoning paths—to balance reasoning depth against verbosity for code generation.

Problem It Solves: Existing tree-search TTS methods suffer from underthinking (shallow reasoning chains that miss problem complexity) and overthinking (verbose chains that waste compute); LogitsCoder addresses both by steering generation quality at the token distribution level rather than through coarse sampling or external verifiers.

How It Works:

  1. Logits Preference Decoding: At each decoding step, adjust the raw logit distribution to upweight tokens that match statistically preferred patterns (e.g., tokens associated with correct reasoning steps in training data), steering the model toward higher-quality token choices without fine-tuning.
  2. Logits Rank Based Path Selection: Generate multiple candidate reasoning chains, then score and rank them using logit-derived confidence signals—selecting paths where the model assigns consistently high probability mass to its own outputs, filtering low-confidence or incoherent chains.
  3. Thoughts Aggregation: Merge the top-ranked diverse reasoning paths into a single coherent chain-of-thought, then decode the final code solution from the aggregated reasoning context.

Architecture:

flowchart LR
    A[Problem Prompt] --> B[Logits Preference Decoding]
    B --> C[N Candidate CoT Paths]
    C --> D[Logits Rank Based Path Selection]
    D --> E[Top-K Paths]
    E --> F[Thoughts Aggregation]
    F --> G[Final Code Output]

Where It’s Useful:

  • Code generation tasks requiring structured multi-step reasoning (e.g., competitive programming, algorithmic problems)
  • Any LLM inference pipeline where test-time compute budget must be controlled without model retraining
  • Scenarios where process reward models are unavailable or too costly to run

Limitations / Caveats:

  • Logit-level preference signals depend on the base model’s calibration—poorly calibrated models may produce misleading confidence scores for path ranking
  • The “statistically preferred patterns” used in Logits Preference Decoding require a reference distribution, implying some offline data collection or profiling step that may not generalize across domains
  • Aggregating diverse paths into a single coherent chain is non-trivial; incoherent merges could degrade rather than improve final output quality

Internal Use Cases:

  • LLM-based ranking/reranking: The logit rank signal is a lightweight proxy for a process reward model—applicable to ranking candidate responses in retrieval-augmented generation pipelines (cf. Monte Carlo Tree Search path scoring) without a separate verifier model
  • Self-consistency extension: Replaces naive majority voting in Self-Consistency Decoding with logit-confidence-weighted path selection, potentially improving answer quality on reasoning-heavy RecSys query understanding tasks
  • Query rewriting diversity control: The overthinking/underthinking framing maps directly to controlling verbosity in chain-of-thought query expansion for retrieval systems
