ConceptMoE: Adaptive Token-to-Concept Compression for Implicit Compute Allocation

ConceptMoE: Adaptive Token-to-Concept Compression for Implicit Compute Allocation

Source: HuggingFace Published: 2026-01-29 Rank: #10 (score: 0.87, max_sim: 0.82) Reading time: ~0 min Tags: Uncategorized

Abstract

Large language models allocate uniform computation across all tokens, ignoring that some sequences are trivially predictable while others require deep reasoning. We introduce ConceptMoE, which dynamically merges semantically similar tokens into concept representations, performing implicit token-level compute allocation. A learnable chunk module identifies optimal boundaries by measuring inter-token similarity, compressing sequences by a target ratio R before they enter the compute-intensive concept model. Crucially, the MoE architecture enables controlled evaluation: we reallocate saved computation to match baseline activated FLOPs (excluding attention map computation) and total parameters, isolating genuine architectural benefits. Under these conditions, ConceptMoE consistently outperforms standard MoE across language and vision-language tasks, achieving +0.9 points on language pretraining, +2.3 points on long context understanding, and +0.6 points on multimodal benchmarks. When converting pretrained MoE during continual training with layer looping, gains reach +5.5 points, demonstrating practical applicability. Beyond performance, ConceptMoE reduces attention computation by up to R^2times and KV cache by Rtimes. At R=2, empirical measurements show prefill speedups reaching 175% and decoding speedups up to 117% on long sequences. The minimal architectural modifications enable straightforward integration into existing MoE, demonstrating that adaptive concept-level processing fundamentally improves both effectiveness and efficiency of large language models.

Key Techniques

  • Dynamic token merging
  • Mixture-of-Experts (MoE) architecture
  • Learnable chunk boundaries
  • Controlled compute reallocation

Learnable Chunk Module (Boundary Detection)

Core Idea: A learned mechanism that identifies optimal token-merging boundaries by measuring semantic similarity between adjacent or nearby tokens, replacing hand-crafted heuristics with differentiable chunking.

Problem It Solves: Deciding which tokens to merge is non-trivial—static rules (e.g., “merge every k tokens”) destroy semantic structure. The chunk module replaces fixed chunking with learned, data-dependent boundaries, enabling the model to adapt merging to actual sequence difficulty and semantic coherence.

How It Works:

  1. Similarity Scoring: For each position or sliding window in the sequence, compute inter-token similarity (e.g., s_i = cosine_similarity(emb_i, emb_{i+1}) or learned scoring function f_θ([emb_i, emb_{i+1}])). This produces a similarity profile across the sequence.
  2. Threshold-Based Segmentation: Apply a learned threshold or differentiable filtering (e.g., top-P selection of low-similarity boundaries) to decide chunk boundaries. Boundaries occur at positions where similarity drops below threshold, creating variable-length chunks.
  3. Gradient-Based Optimization: The chunk module is differentiable end-to-end; boundaries adjust during training to maximize downstream task performance while respecting a target compression ratio R. This couples chunking to downstream MoE and task loss.

Architecture:

Where It’s Useful:

  • Adapting chunking to domain-specific patterns (e.g., code has different redundancy than prose)
  • Tasks where token importance varies (e.g., reasoning steps are dense, narration is sparse)
  • Online or continual learning where chunk boundaries refine over time
  • Vision-language scenarios where patch redundancy varies spatially (high redundancy in uniform backgrounds, low in complex regions)

Limitations / Caveats:

  • Boundary detection adds a learnable module with its own parameters; overfitting risk if chunk module is poorly regularized
  • Compression ratio R enforcement (e.g., target N/R chunks) may conflict with data-driven boundaries; requires careful loss balancing
  • Similarity metric choice is crucial—cosine distance may not capture semantic “mergeability” for all modalities
  • Training cost: computing similarity scores at every forward pass adds overhead that must be amortized by MoE FLOP savings
  • Inference-time chunking is deterministic but boundaries may be brittle (small embedding perturbations cause boundary flip)

Internal Use Cases:

  • Query-Document Matching: In retrieval systems, identify “boilerplate” token spans in documents (high inter-token similarity) and compress them into concept tokens before expensive cross-attention with user query, reducing quadratic attention cost
  • LLM Prompt Compression: Automatically merge redundant prompt tokens (e.g., repeated preambles, system messages) into concepts before expensive decoding, reducing KV cache and improving latency for long-prompt scenarios common in few-shot or RAG systems

Controlled Compute Reallocation for Fair MoE Comparison

Core Idea: Reallocate computation saved by token compression to match baseline MoE’s total activated FLOPs (excluding attention) and parameters, enabling isolation of genuine architectural benefits from compression gains.

Problem It Solves: Token compression inherently reduces compute. Without careful accounting, it’s unclear whether ConceptMoE outperforms baselines because of better architectural design or simply because fewer tokens were processed. This technique ensures fair comparison by “spending” savings on additional MoE capacity or deeper expert layers.

How It Works:

  1. Measure Baseline FLOPs: Calculate total activated FLOPs for standard MoE baseline (sum over all tokens and experts; excludes attention map computation which is shared).
  2. Calculate Savings: ConceptMoE processes N/R tokens instead of N through experts, saving (1 - 1/R) × baseline FLOPs.
  3. Reallocate Saved Compute: Add capacity to ConceptMoE model to consume the saved FLOPs:
    • Increase number of experts while keeping tokens-per-expert constant
    • Increase expert hidden dimensions (wider FFN layers)
    • Increase number of MoE layers with lower activation ratio
    • Ensure total parameters and activated parameters match or exceed baseline
  4. Retest: Evaluate ConceptMoE with reallocated capacity against baseline, ensuring both models have equal compute budgets.

Where It’s Useful:

  • Benchmarking new LLM architectures to avoid conflating compression with actual improvements
  • Internal ML research where fair compute accounting is critical for publication and reproducibility
  • Comparing inference-time vs. train-time optimizations (e.g., layer looping benefits isolated from compression)
  • Any scenario where token count and compute are inversely linked and must be decoupled for honest evaluation

Limitations / Caveats:

  • Compute reallocation is approximate; attention and other components may have different scaling characteristics, making exact FLOPs matching impossible
  • Reallocated capacity (e.g., wider experts) may not be equally beneficial as original architecture; compute is not perfectly fungible
  • Does not account for memory bandwidth or cache effects, which vary with token sequence length—FLOPs alone is incomplete
  • Requires careful engineering to avoid confounding variables; easy to accidentally introduce unequal architectural changes

Internal Use Cases:

  • Ranking Model Upgrades: When proposing a new ranking architecture with compression, ensure comparison against baseline with equal activated capacity (e.g., wider hidden layers or more experts) to isolate true ranking quality gains
  • Production LLM Evaluation: Before deploying efficiency optimizations, enforce that served model improvements are genuine signal, not artifacts of reduced token processing, enabling confident decision-making on compute allocation
