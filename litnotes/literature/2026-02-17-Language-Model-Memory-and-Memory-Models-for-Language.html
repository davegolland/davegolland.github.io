Language Model Memory and Memory Models for Language

Source: arXiv:cs.CL Published: 2026-02-13 Rank: #2 (score: 0.99, max_sim: 0.82) Reading time: ~20 min Tags: Embeddings, LLM, RAG

Abstract

The ability of machine learning models to store input information in hidden layer vector embeddings, analogous to the concept of `memory’, is widely employed but not well characterized. We find that language model embeddings typically contain relatively little input information regardless of data and compute scale during training. In contrast, embeddings from autoencoders trained for input regeneration are capable of nearly perfect memory formation. The substitution of memory embeddings for token sequences leads to substantial computational efficiencies, motivating the introduction of a parallelizable encoder-decoder memory model architecture. Upon causal training these models contain information-poor embeddings incapable of arbitrary information access, but by combining causal and information retention objective functions they learn to form and decode information-rich memories. Training can be further streamlined by freezing a high fidelity encoder followed by a curriculum training approach where decoders first learn to process memories and then learn to additionally predict next tokens. We introduce the perspective that next token prediction training alone is poorly suited for accurate memory formation as the objective itself is non-invertible, motivating the use of combined objective functions for models where the entire input is not exposed.

Causal Encoder-Decoder Memory Model (CEDMM)

Core Idea: A parallelizable encoder-decoder architecture trained with combined causal language modeling and information-retention objectives, enabling language models to form and decode information-rich memory embeddings instead of relying solely on next-token prediction.

Problem It Solves: Standard causal language model training produces information-poor embeddings because next-token prediction is a non-invertible objective — the model never needs to recover full input content from a single embedding, so embeddings compress poorly and cannot serve as faithful memory representations.

How It Works:

  1. Encoder pre-training: Train an autoencoder purely on input reconstruction (regeneration objective) to produce high-fidelity dense embeddings that capture nearly complete input information.
  2. Freeze encoder, train decoder (Stage 1): Fix the encoder weights; train the decoder to process and reconstruct content from memory embeddings — teaching it to “read” memories before learning prediction.
  3. Curriculum fine-tuning (Stage 2): Extend decoder training with a combined objective: information retention loss (reconstruct input from embedding) + causal next-token prediction loss, so the model learns both memory fidelity and generation.

Architecture:

flowchart LR
    T[Token Sequence] --> E[Frozen Encoder]
    E --> M[Memory Embedding]
    M --> D[Decoder]
    D --> R[Reconstructed Input\n+ Next Token Prediction]
    subgraph Combined Objective
        R --> L1[Reconstruction Loss]
        R --> L2[Causal LM Loss]
    end

Where It’s Useful:

  • Long-context compression: replacing token sequences with compact memory embeddings reduces KV-cache and attention costs
  • Retrieval-augmented generation: storing retrieved passages as memory embeddings rather than raw tokens
  • Efficient inference over repeated or cached contexts

Limitations / Caveats:

  • Causal training alone degrades embedding informativeness; combined objectives add training complexity and require tuning loss weighting
  • The frozen-encoder assumption means encoder quality is a hard ceiling on memory fidelity
  • Curriculum staging adds implementation overhead and requires careful scheduling between reconstruction and prediction phases

Internal Use Cases:

  • Ranking/RecSys: User session histories could be compressed into memory embeddings rather than truncated token sequences, preserving more behavioral signal within fixed compute budgets
  • LLM serving: High-traffic repeated system prompts or product catalogs encoded once as memory embeddings, decoded on demand — reducing prefill cost substantially
  • RAG pipelines: Replace chunked passage tokens with encoder-produced memory embeddings as a more compute-efficient retrieval unit, especially relevant when combined with sparse autoencoder interpretability work for debugging what information is retained
