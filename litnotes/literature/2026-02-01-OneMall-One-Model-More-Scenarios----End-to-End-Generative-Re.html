OneMall: One Model, More Scenarios — End-to-End Generative Recommender Family at Kuaishou E-Commerce

Source: arXiv:cs.IR Published: 2026-01-29 Rank: #4 (score: 0.96, max_sim: 0.84) Reading time: ~0 min Tags: Uncategorized

Abstract

In the wave of generative recommendation, we present OneMall, an end-to-end generative recommendation framework tailored for e-commerce services at Kuaishou. Our OneMall systematically unifies the e-commerce’s multiple item distribution scenarios, such as Product-card, short-video and live-streaming. Specifically, it comprises three key components, aligning the entire model training pipeline to the LLM’s pre-training/post-training: (1) E-commerce Semantic Tokenizer: we provide a tokenizer solution that captures both real-world semantics and business-specific item relations across different scenarios; (2) Transformer-based Architecture: we largely utilize Transformer as our model backbone, e.g., employing Query-Former for long sequence compression, Cross-Attention for multi-behavior sequence fusion, and Sparse MoE for scalable auto-regressive generation; (3) Reinforcement Learning Pipeline: we further connect retrieval and ranking models via RL, enabling the ranking model to serve as a reward signal for end-to-end policy retrieval model optimization. Extensive experiments demonstrate that OneMall achieves consistent improvements across all e-commerce scenarios: +13.01% GMV in product-card, +15.32% Orders in Short-Video, and +2.78% Orders in Live-Streaming. OneMall has been deployed, serving over 400 million daily active users at Kuaishou.

Key Techniques

  • E-commerce Semantic Tokenizer
  • Transformer-based Architecture with Query-Former and Sparse MoE
  • Reinforcement Learning for end-to-end retrieval-ranking optimization
  • Multi-behavior sequence fusion with Cross-Attention

Query-Former for Long Sequence Compression

Core Idea: A Transformer-based module that compresses long user interaction sequences into compact query representations while preserving critical behavioral patterns for downstream ranking.

Problem It Solves: User interaction histories can be extremely long (thousands of clicks/views), making full-sequence attention computationally prohibitive. Query-Former enables efficient Transformer-based generation by reducing sequence length while retaining predictive information.

How It Works:

  1. Input: Long user sequence S = [e₁, e₂, …, eₜ] where t can be very large
  2. Project sequence elements to embeddings and apply positional encodings
  3. Use learnable query tokens (similar to BERT’s [CLS] token) to attend over the entire sequence via multi-head self-attention
  4. Output: Compressed representation (fixed-size query vector) that summarizes the most salient behavioral patterns
  5. Pass compressed query to downstream Transformer layers for ranking/generation

Architecture:

flowchart LR
    A["User Sequence<br/>[e₁, e₂, ..., eₜ]"] --> B["Embedding +<br/>Positional Encoding"]
    B --> C["Multi-Head<br/>Attention<br/>Query Tokens"]
    C --> D["Compressed Query<br/>Vector"]
    D --> E["Downstream<br/>Ranking/Generation"]

Where It’s Useful:

  • Long-context recommendation (users with extensive interaction history)
  • Reducing computational cost of attention in generative recommendation
  • Attention bottleneck in sequential recommendation systems

Limitations / Caveats:

  • Information loss due to compression; may discard low-frequency but important signals
  • Fixed compression ratio may not be optimal across different user types
  • Performance depends on quality of learnable query initialization

Internal Use Cases:

  • Compressing dense user sequences before ranking model inference
  • Enabling multi-behavior fusion (see below) without exploding computational cost

Cross-Attention for Multi-Behavior Sequence Fusion

Core Idea: Cross-attention mechanism that dynamically fuses multiple user behavior sequences (clicks, purchases, shares, etc.) by allowing one behavior type to attend to others, creating a unified behavioral representation.

Problem It Solves: Users exhibit multiple types of behaviors (implicit: views/clicks, explicit: purchases/likes, social: shares/follows). Treating these independently loses correlations; simple concatenation is inefficient. Cross-attention learns to weight and integrate heterogeneous behaviors.

How It Works:

  1. Encode each behavior type separately: Click sequence C, Purchase sequence P, Share sequence S, etc., each producing embeddings
  2. For each behavior type, apply cross-attention where:
    • Query comes from one behavior (e.g., clicks): Q = W_q · C
    • Key and Value come from another behavior (e.g., purchases): K = W_k · P, V = W_v · P
  3. Compute multi-head attention: Attention(Q, K, V) = softmax(QK^T/√d)V
  4. Concatenate or sum outputs from all cross-attention pairs to create fused representation
  5. Pass fused multi-behavior embedding to ranking/generation modules

Architecture:

flowchart LR
    A["Click<br/>Sequence"] --> Q["Query"]
    B["Purchase<br/>Sequence"] --> KV["Key, Value"]
    C["Share<br/>Sequence"] --> KV2["Key, Value"]
    Q --> CA["Cross-Attention<br/>Modules"]
    KV --> CA
    KV2 --> CA
    CA --> D["Fused Behavioral<br/>Representation"]

Where It’s Useful:

  • Multi-behavior recommendation systems (clicks + purchases + shares)
  • Learning inter-behavior dependencies (e.g., how clicks predict purchases)
  • Implicit-explicit behavior fusion in e-commerce

Limitations / Caveats:

  • Requires careful behavior sequence alignment (different timestamps/frequencies)
  • Quadratic complexity in sequence length; needs compression (Query-Former) to scale
  • Attention weights may be dominated by high-frequency behaviors if not properly normalized

Internal Use Cases:

  • Fusing multi-stage conversion behaviors (view → click → purchase → review)
  • Balancing weak signals (shares) with strong signals (purchases) in unified representation

Sparse Mixture-of-Experts (MoE) for Auto-Regressive Generation

Core Idea: A scalable auto-regressive generation architecture using Sparse MoE where a router network dynamically selects a small subset of expert modules for each token prediction, enabling efficient scaling without dense computation.

Problem It Solves: Generative recommendation requires predicting long sequences of item tokens auto-regressively. Naïve dense Transformer scaling is computationally prohibitive at Kuaishou’s scale (400M DAU). Sparse MoE enables parameter-efficient generation by activating only a subset of capacity per step.

How It Works:

  1. Decoder Transformer receives previous predicted tokens: [item₁, item₂, …, itemₙ₋₁]
  2. At each decoding step, embed current context and pass through Transformer layers
  3. For each Transformer layer’s FFN (feed-forward network), replace dense FFN with sparse MoE:
    • Router network: r = softmax(W_router · h) produces logits over K experts
    • Top-k selection: Select top-k experts (e.g., k=2) with highest router scores
    • Expert computation: Only selected experts process the token: output = Σᵢ∈selected rᵢ · Expertᵢ(h)
  4. Aggregate expert outputs and continue to next layer
  5. Final layer outputs logits over item vocabulary; sample or argmax to select next item token
  6. Repeat until end-of-sequence token or maximum length

Architecture:

flowchart LR
    A["Predicted Tokens<br/>[item₁, ..., itemₙ₋₁]"] --> B["Embedding +<br/>Positional Encoding"]
    B --> C["Transformer<br/>Layers"]
    C --> D["Sparse MoE<br/>Router"]
    D --> E["Select Top-k<br/>Experts"]
    E --> F["Expert<br/>Aggregation"]
    F --> C
    C --> G["Output Logits<br/>over Vocabulary"]
    G --> H["Sample/Argmax<br/>Next Token"]
    H --> I["Next Step"]

Where It’s Useful:

  • Large-scale auto-regressive recommendation at production scale (400M+ users)
  • Item recommendation with long recommendation lists
  • Multi-scenario item prediction (product + video + stream items)

Limitations / Caveats:

  • Requires careful load balancing to prevent expert imbalance (some experts inactive, others overloaded)
  • Router training can be unstable; may need auxiliary loss (e.g., load balancing auxiliary loss)
  • Top-k selection is non-differentiable; requires straight-through estimator or gating tricks
  • Increased model complexity and training overhead compared to dense models

Internal Use Cases:

  • Scaling recommendation models to production scale without proportional compute increase
  • Multi-task generation where different item types route to specialized experts

Reinforcement Learning Pipeline for End-to-End Retrieval-Ranking Optimization

Core Idea: A reinforcement learning framework that treats the ranking model as a reward signal, enabling the retrieval (candidate generation) model to optimize directly for downstream ranking and conversion metrics end-to-end, rather than optimizing retrieval and ranking separately.

Problem It Solves: Traditional recommendation decouples retrieval (candidate generation) from ranking. Retrieval is optimized for recall/coverage, ranking for CTR/conversion, but retrieved candidates may not be optimal for the downstream ranker’s objective. This two-stage separation leaves optimization gaps. RL bridges this by making retrieval learn to fetch candidates that the ranker scores highly.

How It Works:

  1. Policy (Retrieval Model): πθ(a|s) outputs candidate item distributions given user state s, parameterized by Sparse MoE generative model trained as auto-regressive sequence
  2. Reward Signal (Ranking Model): Frozen or slowly-updated ranker R(s, a) scores each retrieved candidate item a, producing reward r = R(s, a) (can be CTR prediction, conversion probability, or GMV proxy)
  3. Trajectory Rollout: Sample K retrieval trajectories (sequences of retrieved items) from policy πθ
  4. Return Computation: Compute discounted cumulative reward: G_t = Σᵢ₌ₜ^T γⁱ⁻ᵗ r(sᵢ, aᵢ) for each step
  5. Policy Gradient Update: ∇J(θ) = E[∇log πθ(a|s) · G_t], updating retrieval model to maximize expected ranking reward
  6. Iteration: Periodically retrain ranking model on new retrieval distribution, creating feedback loop

Architecture:

flowchart LR
    A["User State<br/>s"] --> B["Retrieval Policy<br/>πθ"]
    B --> C["Sampled Items<br/>{a₁, a₂, ...}"]
    C --> D["Ranking Model<br/>R s,a"]
    D --> E["Reward Signal<br/>r = R s,a"]
    E --> F["Policy Gradient<br/>Update θ"]
    F --> B
    D --> G["Ranking Training<br/>Loop"]
    G --> D

Where It’s Useful:

  • End-to-end recommendation optimization where retrieval and ranking must align
  • Multi-objective optimization (GMV, orders, user satisfaction) as reward signals
  • Production recommendation systems where early-stage decisions (retrieval) impact later stages

Limitations / Caveats:

  • Reward signal (ranking model) must be well-calibrated and stable, or RL will optimize for its errors
  • High variance in policy gradient estimates may require large batch sizes or variance reduction techniques (baseline, importance sampling)
  • Computational cost is significant (requires sampling trajectories, ranking each candidate, backprop through policy)
  • Training instability if reward and policy diverge too far; requires careful learning rate scheduling

Internal Use Cases:

  • End-to-end optimization of candidate generation pipelines in RecSys
  • Aligning multi-stage ranking funnel (retrieval → ranking → re-ranking) via shared objective
  • Online learning / contextual bandit formulation for adaptive retrieval strategies
