When RL Meets Adaptive Speculative Training: A Unified Training-Serving System

Source: arXiv:cs.LG Published: 2026-02-06 Rank: #1 (score: 0.97, max_sim: 0.84) Reading time: ~0 min Tags: Speculative-Decoding

Abstract

Speculative decoding can significantly accelerate LLM serving, yet most deployments today disentangle speculator training from serving, treating speculator training as a standalone offline modeling problem. We show that this decoupled formulation introduces substantial deployment and adaptation lag: (1) high time-to-serve, since a speculator must be trained offline for a considerable period before deployment; (2) delayed utility feedback, since the true end-to-end decoding speedup is only known after training and cannot be inferred reliably from acceptance rate alone due to model-architecture and system-level overheads; and (3) domain-drift degradation, as the target model is repurposed to new domains and the speculator becomes stale and less effective. To address these issues, we present Aurora, a unified training-serving system that closes the loop by continuously learning a speculator directly from live inference traces. Aurora reframes online speculator learning as an asynchronous reinforcement-learning problem: accepted tokens provide positive feedback, while rejected speculator proposals provide implicit negative feedback that we exploit to improve sample efficiency. Our design integrates an SGLang-based inference server with an asynchronous training server, enabling hot-swapped speculator updates without service interruption. Crucially, Aurora supports day-0 deployment: a speculator can be served immediately and rapidly adapted to live traffic, improving system performance while providing immediate utility feedback. Across experiments, Aurora achieves a 1.5x day-0 speedup on recently released frontier models (e.g., MiniMax M2.1 229B and Qwen3-Coder-Next 80B). Aurora also adapts effectively to distribution shifts in user traffic, delivering an additional 1.25x speedup over a well-trained but static speculator on widely used models (e.g., Qwen3 and Llama3).

Key Techniques

  • Asynchronous reinforcement learning
  • Live inference trace feedback
  • Hot-swapped speculator updates

Aurora: RL-Based Online Speculative Decoding

Core Idea: A unified training-serving system that continuously learns and updates a speculative decoder directly from live inference traces using reinforcement learning, enabling day-0 deployment and runtime adaptation to distribution shifts.

Problem It Solves: Traditional speculative decoding separates training from serving, causing (1) high time-to-serve since speculators must be fully trained offline before deployment, (2) delayed utility feedback—true speedup is only known after training completes, and (3) domain-drift degradation when the target LLM is repurposed to new domains.

How It Works:

  1. Immediate Deployment: Deploy an initial speculator (even untrained) alongside the target LLM to start serving traffic immediately
  2. Live Trace Collection: During inference, collect accepted and rejected draft tokens from the speculative decoding process
  3. Asynchronous RL Training: Frame speculator learning as an RL problem where accepted tokens provide positive rewards and rejected proposals provide negative feedback; train the speculator asynchronously on these traces
  4. Hot-Swap Updates: Periodically update the serving speculator with improved checkpoints without service interruption
  5. Continuous Adaptation: As traffic patterns or target LLM domains shift, the speculator automatically adapts through ongoing learning

Architecture:

flowchart LR
    A[Inference Traffic] --> B[SGLang Serving]
    B --> C[Draft Acceptance/Rejection]
    C --> D[Trace Buffer]
    D --> E[Async RL Trainer]
    E --> F[Updated Speculator]
    F -.Hot Swap.-> B
    C --> G[Accelerated Generation]

Where It’s Useful:

  • Production LLM serving where target models evolve or get repurposed across domains
  • Scenarios requiring immediate deployment without long offline training periods
  • Multi-tenant or multi-domain serving where traffic distributions vary over time
  • Frontier models where pre-training domain-specific speculators offline is impractical

Limitations / Caveats:

  • Requires infrastructure to support asynchronous training and hot-swapping of models during serving
  • Initial day-0 performance depends on the quality of the starting speculator (though any speculator can be deployed)
  • RL training efficiency depends on sufficient traffic volume to collect diverse acceptance/rejection signals
  • System complexity increases compared to static speculative decoding

Internal Use Cases:

  • LLM Serving: Deploy new LLMs to production immediately with adaptive speculation, eliminating the offline training bottleneck for each new model variant
  • Multi-Domain RecSys/Ranking: Apply the RL-from-feedback paradigm to continuously adapt lightweight draft models for candidate generation in recommendation systems as user behavior shifts
  • A/B Testing: Rapidly iterate on speculator architectures by deploying candidates and measuring true end-to-end latency metrics, not just proxy metrics like acceptance rate
