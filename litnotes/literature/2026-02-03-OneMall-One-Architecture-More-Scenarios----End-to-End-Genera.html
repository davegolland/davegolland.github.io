OneMall: One Architecture, More Scenarios — End-to-End Generative Recommender Family at Kuaishou E-Commerce

Source: arXiv:cs.IR Published: 2026-01-29 Rank: #9 (score: 0.95, max_sim: 0.84) Reading time: ~0 min Tags: Uncategorized Code: https://arxiv.org/pdf/2601.21770v2

Abstract

In the wave of generative recommendation, we present OneMall, an end-to-end generative recommendation framework tailored for e-commerce services at Kuaishou. Our OneMall systematically unifies the e-commerce’s multiple item distribution scenarios, such as Product-card, short-video and live-streaming. Specifically, it comprises three key components, aligning the entire model training pipeline to the LLM’s pre-training/post-training: (1) E-commerce Semantic Tokenizer: we provide a tokenizer solution that captures both real-world semantics and business-specific item relations across different scenarios; (2) Transformer-based Architecture: we largely utilize Transformer as our model backbone, e.g., employing Query-Former for long sequence compression, Cross-Attention for multi-behavior sequence fusion, and Sparse MoE for scalable auto-regressive generation; (3) Reinforcement Learning Pipeline: we further connect retrieval and ranking models via RL, enabling the ranking model to serve as a reward signal for end-to-end policy retrieval model optimization. Extensive experiments demonstrate that OneMall achieves consistent improvements across all e-commerce scenarios: +13.01% GMV in product-card, +15.32% Orders in Short-Video, and +2.78% Orders in Live-Streaming. OneMall has been deployed, serving over 400 million daily active users at Kuaishou.

Key Techniques

  • E-commerce Semantic Tokenizer
  • Query-Former for sequence compression
  • Reinforcement Learning Pipeline

Query-Former for Long Sequence Compression

Core Idea: A Transformer-based module that compresses long user behavior sequences into fixed-length query representations by learning to aggregate the most relevant historical interactions into compact learnable tokens.

Problem It Solves: User behavior sequences in e-commerce can span hundreds of interactions, creating computational bottlenecks for Transformer models which have O(n²) complexity. Direct processing of full sequences is infeasible, yet truncation or simple pooling loses important behavioral context critical for accurate recommendations.

How It Works:

  1. Input: Raw user behavior sequence B = [b₁, b₂, …, bₙ] where n is potentially large (hundreds of interactions)
  2. Initialize a fixed set of K learnable query tokens Q = [q₁, q₂, …, qₖ] (typically K << n)
  3. Apply cross-attention from query tokens to the full behavior sequence: Attention(Q, B, B) = softmax(QBᵀ/√d)Bᵥ
  4. Output: Compressed representation C ∈ ℝᵏˣᵈ that captures essential behavioral patterns in fixed size
  5. Pass C to downstream recommendation modules (retrieval/ranking)

Architecture:

flowchart LR
    A["Long Behavior Sequence<br/>b₁...bₙ"] --> B["Cross-Attention<br/>from Query Tokens"]
    C["Learnable Queries<br/>q₁...qₖ"] --> B
    B --> D["Compressed Representation<br/>K tokens"]
    D --> E["Downstream<br/>Recommendation"]

Where It’s Useful:

  • Compressing long user interaction histories in sequential recommendation
  • Handling users with extensive browsing/purchase records without sequence truncation
  • Real-time inference on mobile/edge where latency is critical
  • Scaling Transformer-based rankers to production recommendation pipelines

Limitations / Caveats:

  • Compression ratio K/n must be carefully tuned; too aggressive compression loses signal
  • Query initialization and training dynamics affect convergence
  • Cross-attention is still O(Kn), so gains depend on compression rate relative to full O(n²)
  • May lose tail interactions that could be important for niche recommendations

Internal Use Cases:

  • User encoder in two-tower retrieval models handling long-range dependencies
  • Replacing fixed-size RNN hidden states with interpretable compressed query representations
  • Pre-processing for MoE routing (next section) to handle variable sequence lengths efficiently

Cross-Attention for Multi-Behavior Sequence Fusion

Core Idea: A cross-attention mechanism that fuses multiple user behavior sequences (e.g., clicks, purchases, shares, likes) into a unified context representation by learning fine-grained interactions between different behavior types.

Problem It Solves: Users exhibit diverse behaviors in e-commerce (clicking, adding to cart, purchasing, sharing) that carry different semantic meanings and predictive signals. Naive concatenation or simple fusion loses the rich signal from cross-behavior dependencies (e.g., a purchased item’s relationship to previously clicked items). Multi-task or multi-head approaches lack explicit interaction modeling.

How It Works:

  1. Input: M behavior sequences S = [S₁, S₂, …, Sₘ] where Sᵢ represents one behavior type (clicks, purchases, etc.)
  2. Embed each sequence: Eᵢ = Embed(Sᵢ) ∈ ℝⁿⁱˣᵈ (varying lengths nᵢ allowed)
  3. Apply cross-attention between pairs of behavior types: For each i,j: CrossAtt(Eᵢ, Eⱼ) = softmax(EᵢWQ·EⱼWKᵀ/√d)EⱼWV
  4. Learn implicit behavior relationships (e.g., which clicks led to purchases, which likes precede clicks)
  5. Output: Fused representation F ∈ ℝᵈ capturing all M behavior signals and their interactions

Architecture:

flowchart LR
    A["Click Sequence"] --> B["Embedding"]
    C["Purchase Sequence"] --> D["Embedding"]
    E["Like Sequence"] --> F["Embedding"]
    B --> G["Cross-Attention<br/>Fusion"]
    D --> G
    F --> G
    G --> H["Unified Multi-Behavior<br/>Representation"]

Where It’s Useful:

  • Multi-behavior recommendation (clicks, purchases, shares, comments, etc.)
  • Capturing behavior hierarchies (e.g., clicks → cart adds → purchases)
  • Sequential recommendation with heterogeneous feedback signals
  • Context modeling in ranking systems with diverse user signals

Limitations / Caveats:

  • Behavior sequences may have highly imbalanced lengths (many clicks vs. few purchases)
  • Cross-attention learns pairwise interactions; fully modeling M behavior types requires O(M²) attention computations
  • Interpretation of learned cross-behavior weights can be opaque
  • Behavior type embeddings must be compatible for meaningful cross-attention

Internal Use Cases:

  • User context encoder in retrieval models fusing implicit and explicit feedback
  • Multi-signal ranking features replacing hand-crafted behavior aggregation
  • Foundation for the RL reward signal (next section) which optimizes over fused behavior predictions

Sparse Mixture of Experts (MoE) for Auto-Regressive Generation

Core Idea: A scalable auto-regressive token generation architecture using Sparse MoE that routes each position’s generation decision to specialized expert sub-networks, enabling efficient scaling without dense computation across all parameters.

Problem It Solves: Generative recommendation requires auto-regressive token generation (producing ranking/retrieval decisions sequentially), but dense Transformer scaling becomes prohibitively expensive. Full Transformer decoding for each token position is O(d²) per step. Simple Transformer baselines cannot handle the parameter scale needed for modeling complex e-commerce scenarios across all 400M+ users.

How It Works:

  1. Input: Compressed user representation U and previous generated tokens [t₁, t₂, …, tᵢ₋₁]
  2. At each generation step i, compute: hᵢ = Attention(U, [t₁, …, tᵢ₋₁])
  3. Compute routing logits: rᵢ = Router(hᵢ) ∈ ℝᴱ (E = number of experts)
  4. Select top-k experts via sparse gating: gᵢ = TopK(softmax(rᵢ)) (typically k=2)
  5. Route hidden state to selected experts: oᵢ = Σⱼ∈TopK gᵢⱼ·Expertⱼ(hᵢ)
  6. Generate next token: P(tᵢ|t₁…tᵢ₋₁,U) = softmax(oᵢWtoken)
  7. Repeat for auto-regressive decoding until stopping token

Architecture:

flowchart LR
    A["User Representation<br/>+ Previous Tokens"] --> B["Attention Encoder"]
    B --> C["Router Network"]
    C --> D["Sparse Gating<br/>Top-k Selection"]
    E["Expert Pool<br/>E₁...Eₙ"] --> D
    D --> F["Expert Computation<br/>Weighted Sum"]
    F --> G["Token Generation<br/>softmax"]
    G --> H["Next Token"]

Where It’s Useful:

  • Large-scale auto-regressive recommendation ranking/retrieval
  • Generative recommendation systems requiring dense parameter coverage with sparse computation
  • Scaling recommendation models to millions of items/users without proportional compute cost
  • Multi-scenario recommendation where different scenarios route through specialized experts

Limitations / Caveats:

  • Sparse routing can cause load balancing issues (some experts overused, others underutilized)
  • Training instability: router gradients are discrete (hard routing) or biased (soft routing with gumbel-softmax)
  • Communication overhead on distributed systems if experts are on different GPUs/TPUs
  • Top-k selection destroys gradient flow to non-selected experts during training
  • Expert specialization may not emerge without auxiliary balancing losses

Internal Use Cases:

  • Auto-regressive ranking model where each step generates next item to rank
  • Retrieval candidate generation that sequentially produces items
  • Leveraging domain expertise: different experts for different scenarios (product-card expert, short-video expert, live-streaming expert)
  • Connection point for RL optimization (next section) where MoE outputs enable discrete policy actions

Reinforcement Learning Pipeline for End-to-End Retrieval-Ranking Optimization

Core Idea: An RL framework that connects retrieval (candidate generation) and ranking models, enabling the ranking model’s predictions to serve as reward signals that directly optimize the retrieval policy end-to-end.

Problem It Solves: Traditional recommendation pipelines (retrieval → ranking) optimize components independently. Retrieval models maximize recall of diverse candidates; ranking models maximize relevance. This decomposition misses the downstream impact: retrieved candidates that ranking subsequently downranks are wasted computation. No mechanism propagates ranking quality back to improve retrieval, leading to retrieval-ranking mismatch and suboptimal GMV/conversion outcomes.

How It Works:

  1. State: User context state S = {user behavior, session context, scenario type}
  2. Retrieval Policy: π_retrieval(a|S) generates candidate set A = {item₁, item₂, …, itemₖ} of size k through sparse MoE auto-regressive generation (previous section)
  3. Ranking Oracle: Score candidates: R = rank_model(A, S), where rank_model outputs relevance scores or predicted CTR/conversion
  4. Reward Signal: Define reward r(a) based on ranking output:
    • r(a) = predicted_CTR(a) for short-video scenario
    • r(a) = predicted_conversion_rate(a) for product-card scenario
    • r(a) = weighted combination for multi-objective (GMV-aware)
  5. RL Objective: Optimize retrieval policy via policy gradient: ∇J = 𝔼[∇log π_retrieval(a|S) · r(a)]
  6. End-to-End Training: Backprop ranking rewards through retrieval via reinforce/actor-critic algorithms

Architecture:

flowchart LR
    A["User State S"] --> B["Retrieval Policy<br/>Auto-Regressive MoE"]
    B --> C["Candidate Set A"]
    C --> D["Ranking Model<br/>Score Candidates"]
    D --> E["Reward Signal<br/>CTR/Conversion"]
    E --> F["Policy Gradient<br/>Optimization"]
    F -->|Backprop| B

Where It’s Useful:

  • End-to-end optimization of multi-stage recommendation systems
  • Learning what types of candidates to retrieve based on actual ranking quality
  • Handling exploration-exploitation in retrieval (RL naturally balances this)
  • Multi-objective optimization (GMV, orders, engagement) where ranking encodes objectives
  • Aligning business metrics (orders, GMV) directly into the retrieval learning signal

Limitations / Caveats:

  • Requires online interaction data or offline RL formulation; off-policy training is sample-inefficient
  • Reward signal design is crucial; sparse/delayed rewards (purchase signal) require credit assignment
  • Policy gradient methods have high variance; requires variance reduction (baseline, importance weighting)
  • Ranking model must be differentiable or reward must be treated as blackbox (policy gradient only)
  • Can amplify ranking biases back into retrieval (if ranker has systematic biases, RL learns to exploit them)
  • Computational cost: must score full candidate set A with ranking model every step

Internal Use Cases:

  • Learning retrieval strategies that maximize downstream ranking utility (not just diversity/coverage)
  • Contextual bandit formulation for exploration in retrieval
  • Connection to business KPIs: reward shaped from GMV/order metrics rather than engagement proxies
  • Multi-scenario policy learning: single RL framework where reward definition varies by scenario (live-streaming vs. product-card)

Analysis: OneMall’s Position in the Generative Recommender Landscape

OneMall extends the foundational work of OneRec and the broader generative recommendation paradigm by addressing a critical challenge that prior work acknowledged but didn’t fully solve: scenario fragmentation in e-commerce. While “The Rise of Generative Recommenders” highlighted how traditional multi-stage systems suffer from fragmentation across retrieval, ranking, and value modeling stages, OneMall tackles a different but equally important fragmentation—distributing recommendations across heterogeneous e-commerce scenarios (product cards, short-video, live-streaming). The paper bridges this gap by introducing an E-commerce Semantic Tokenizer that captures both semantic content and business-specific item relations across scenarios, effectively creating a unified representation layer. This is a natural evolution of the generative paradigm: if one unified model can replace the traditional three-stage pipeline, then one unified tokenizer can bridge multiple distribution channels—a scaling principle aligned with the 2025 research trends around the “One” series models.

The technical architecture itself represents a sophisticated synthesis of LLM techniques applied to e-commerce constraints. OneMall employs Query-Former for long sequence compression and Sparse MoE for scalable auto-regressive generation—both techniques that mirror advances in LLM efficiency. The Cross-Attention mechanism for multi-behavior sequence fusion suggests a deeper insight: recommendation systems now operate on sequences of user behaviors much like language models operate on token sequences. However, OneMall’s innovation lies in its Reinforcement Learning Pipeline, which connects retrieval and ranking models where the ranking model serves as a reward signal. This represents a methodological advance beyond OneRec’s basic end-to-end training: it creates a closed-loop optimization where downstream ranking performance directly informs upstream retrieval policy—solving a key tension in generative recommendation where you must trade off between diversity (retrieval) and conversion (ranking).

The empirical results validate this architectural philosophy across all scenarios, with particularly strong improvements in structured product recommendations (+13.01% GMV), demonstrating that unified generative models can indeed handle the business-critical metrics that matter in real e-commerce operations. The scale of deployment—400 million daily active users—confirms that generative recommendation has moved from research prototype to production reality, but OneMall’s contribution is specifically showing that one architecture can serve multiple scenarios without degradation, which directly challenges the traditional assumption that different distribution channels require different models.
