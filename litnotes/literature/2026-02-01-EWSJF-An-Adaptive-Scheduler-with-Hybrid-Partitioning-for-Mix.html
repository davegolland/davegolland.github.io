EWSJF: An Adaptive Scheduler with Hybrid Partitioning for Mixed-Workload LLM Inference

Source: arXiv:cs.AI Published: 2026-01-29 Rank: #12 (score: 0.97, max_sim: 0.84) Reading time: ~20 min Tags: RAG, LLM Code: https://anonymous.4open.science/r/vllm_0110-32D8

Abstract

Serving Large Language Models (LLMs) under mixed workloads—short, latency-sensitive interactive queries alongside long, throughput-oriented batch requests—poses a fundamental scheduling challenge. Standard First-Come, First-Served (FCFS) policies suffer from severe head-of-line blocking, leading to high tail latency and underutilized hardware. We introduce EWSJF (Effective Workload-based Shortest Job First), an adaptive request-level scheduler that learns workload structure in real time to jointly improve fairness and throughput. EWSJF operates upstream of execution-level schedulers and integrates four components: (1) Refine-and-Prune, an unsupervised partitioning algorithm that discovers performance-homogeneous request groups; (2) Dynamic Queue Routing for assigning requests to these groups; (3) Density-Weighted Scoring, a context-aware prioritization function balancing urgency and fairness; and (4) Bayesian Meta-Optimization, which continuously tunes scoring and partitioning parameters based on live performance feedback. Implemented in vLLM, EWSJF improves end-to-end throughput by over 30% and reduces average Time-To-First-Token for short requests by up to 4x compared to FCFS. These results demonstrate that adaptive, learning-based request scheduling is a critical missing layer for efficient and responsive LLM serving. Implementation available at https://anonymous.4open.science/r/vllm_0110-32D8.

Dynamic Queue Routing

Core Idea: A real-time request assignment mechanism that routes incoming requests to appropriate partition queues based on learned workload characteristics and current queue state.

Problem It Solves: Once requests are partitioned into groups, the scheduler must decide which queue each new request should join. Naive routing (e.g., round-robin) ignores request-specific properties and queue dynamics, leading to imbalanced queues and suboptimal scheduling.

How It Works:

  1. Feature Extraction: Extract request properties at arrival (prompt length L_p, expected token generation length L_g, arrival time t)
  2. Queue State Assessment: For each partition queue i, compute current queue depth, average wait time, and recent throughput
  3. Assignment Decision: Use a learned model (e.g., nearest-neighbor in feature space, lightweight decision tree, or rule-based heuristic) to select the best-matching partition queue for the new request
  4. Adaptive Updates: Periodically retrain or adjust routing rules based on performance feedback (e.g., which queue assignments led to good/poor latency outcomes)

Where It’s Useful:

  • Multi-queue scheduling systems where queue selection is not obvious
  • Online load balancing in heterogeneous server clusters
  • Real-time request routing in microservice architectures

Limitations / Caveats:

  • Routing decisions made at arrival time with incomplete information about future workload
  • Requires maintaining multiple queues, increasing memory and switching overhead
  • Misrouting a request to the wrong queue persists for the duration of execution (no mid-execution migration)
  • Performance depends heavily on the quality of learned routing rules

Internal Use Cases:

  • Search ranking: Routing queries to specialized rankers (neural vs. rule-based) based on query intent
  • LLM prompt routing: Steering requests to specialized model instances (e.g., smaller models for simpler queries)
  • Recommendation filtering: Assigning user sessions to different candidate generation strategies based on session state

Density-Weighted Scoring (DWS)

Core Idea: A context-aware prioritization function that balances request urgency and fairness by combining queue density, wait time, and request characteristics into a single score for scheduling decisions.

Problem It Solves: Fixed priority schemes (e.g., always prioritize short requests) are unfair to long requests, while pure FCFS causes head-of-line blocking. DWS adapts priorities dynamically based on current queue state to maximize responsiveness without starving longer requests.

How It Works:

  1. Urgency Metric: Compute urgency for request r as a function of wait time W_r and request length L_r (e.g., W_r / log(L_r) to normalize by length)
  2. Queue Density Factor: For each partition group, compute density ρ = (queue_depth) / (total_system_capacity) to reflect congestion
  3. Weighted Score: Calculate priority score as Score(r) = (Urgency(r)) × (1 + α × ρ) where α is a learned weight parameter
  4. Scheduling Decision: Select the request with highest score across all queues; if tied, use fairness tiebreaker (oldest request wins)
  5. Parameter Learning: Update α and urgency function coefficients via Bayesian optimization using observed latency outcomes

Where It’s Useful:

  • Request-level scheduling in multi-queue systems
  • Fair prioritization in resource-constrained environments
  • Systems combining SLA-sensitive and batch workloads

Limitations / Caveats:

  • Scoring function is a heuristic; optimal weights depend on workload distribution and hardware characteristics
  • Urgency metric may not capture all relevant request properties (e.g., memory requirements, GPU affinity)
  • High contention scenarios may still cause unfairness if density factor is not tuned correctly
  • Requires careful tuning of α; poor choices lead to starvation or prioritization inversions

Internal Use Cases:

  • RecSys ranking: Weighting candidate items by freshness urgency and ranking quality fairness constraints
  • Feed ranking: Balancing engagement prediction urgency against diversity and fair representation of creators
  • CTR prediction: Adapting model inference priority based on data staleness and batch size pressures

Bayesian Meta-Optimization

Core Idea: A closed-loop learning system that continuously tunes EWSJF’s partitioning and scoring parameters by treating observed performance metrics as feedback signals in a Bayesian optimization framework.

Problem It Solves: EWSJF has multiple hyperparameters (clustering k, urgency weights, density scaling α, partitioning thresholds) that affect performance. Manual tuning is infeasible, and optimal values vary across workloads. Bayesian optimization enables automatic, online parameter tuning without exhaustive grid search.

How It Works:

  1. Observation: Monitor end-to-end performance metrics (average Time-To-First-Token TTFT, throughput, tail latency percentiles) over a sliding window of requests
  2. Objective Function: Define a composite objective J = (1 - λ) × avg_TTFT + λ × throughput, where λ balances latency vs. throughput (λ is also a tuning parameter)
  3. Parameter Hypothesis: Maintain a posterior distribution over promising parameter values using Bayesian inference (e.g., Gaussian Process surrogate model)
  4. Acquisition & Exploration: Select next parameter candidate using an acquisition function (e.g., Expected Improvement) that balances exploiting good parameters and exploring uncertain regions
  5. Feedback Loop: Execute with candidate parameters, collect performance metrics, update posterior belief, and iterate
  6. Online Adaptation: Re-optimize parameters every M requests or when workload characteristics change significantly

Architecture:

flowchart LR
    A["EWSJF Scheduler\nwith Parameters θ"] --> B["Execute Requests"]
    B --> C["Monitor Performance\nMetrics"]
    C --> D["Bayesian Optimization\nUpdate Posterior P(θ|data)"]
    D --> E["Acquisition Function\nSelect Next θ"]
    E --> F{Check\nConvergence}
    F -->|No| A
    F -->|Yes| G["Use Optimized\nParameters"]

Where It’s Useful:

  • Adaptive systems that must tune hyperparameters online
  • Multi-objective optimization in production systems where workload varies
  • Meta-learning frameworks that auto-tune algorithm behavior

Limitations / Caveats:

  • Bayesian optimization has computational overhead; if re-optimized too frequently, cost may exceed scheduling savings
  • Surrogate model (e.g., GP) scales poorly in high-dimensional parameter spaces (curse of dimensionality)
  • Convergence requires sufficient data; early in a session, parameter estimates are noisy
  • Assumes performance metrics are stable enough to guide optimization (high variance metrics reduce signal quality)
  • May get stuck in local optima if objective landscape is multi-modal

Internal Use Cases:

  • RecSys ranking: Auto-tuning diversity weight, freshness penalty, and position bias correction based on online A/B test feedback
  • LLM serving: Optimizing batch size, token prediction lookahead, and scheduler thresholds per workload pattern
  • Search ranking: Adapting neural-to-rule-based ranker balance based on query satisfaction metrics

EWSJF: Integrated Adaptive Request-Level Scheduler

Core Idea: A four-component orchestrated system that combines partitioning discovery, request routing, dynamic prioritization, and online parameter tuning to achieve both fairness and throughput in mixed-workload LLM serving.

Problem It Solves: FCFS scheduling in LLM inference causes severe head-of-line blocking—long batch requests delay short interactive queries, leading to poor tail latency (4x worse) and throughput underutilization. No single fixed policy balances both workload types fairly.

How It Works:

  1. Discover Partitions (Refine-and-Prune): Identify performance-homogeneous request groups from streaming request data
  2. Route Incoming Requests (Dynamic Queue Routing): Assign each new request to the best-matching partition queue
  3. Prioritize Requests (Density-Weighted Scoring): Score all queued requests considering urgency, fairness, and current queue congestion; execute highest-scoring request
  4. Optimize Continuously (Bayesian Meta-Optimization): Monitor performance metrics and auto-tune clustering parameters, urgency weights, and density scaling factors

Architecture:

flowchart TD
    A["Incoming Request Stream"] --> B["Feature Extraction\n(length, tokens, arrival)"]
    B --> C["Refine-and-Prune\nPartitioning"]
    C --> D["Dynamic Queue Routing\nAssign to Queue i"]
    D --> E["Request Queue i"]
    E --> F["Density-Weighted Scoring\nCompute Priority"]
    F --> G["Select Highest-Scoring\nRequest"]
    G --> H["Execute on GPU\nvLLM Engine"]
    H --> I["Monitor Metrics\nTTFT, Throughput, Latency"]
    I --> J["Bayesian Meta-Optimization\nTune Parameters"]
    J --> C

Where It’s Useful:

  • Multi-tenant LLM inference serving (e.g., shared GPU clusters)
  • Systems with mixed latency-sensitive and throughput-oriented workloads
  • Cloud-based language model APIs with heterogeneous user requests
  • Any request scheduling scenario where workload distribution is unpredictable

Limitations / Caveats:

  • Operates at request level, before execution-level schedulers; does not directly optimize token-level scheduling
  • Requires monitoring infrastructure to collect performance feedback for Bayesian optimization
  • Partition quality depends on request feature selection and clustering algorithm
  • Parameter tuning introduces non-determinism; exact reproducibility is difficult
  • Overhead of maintaining multiple queues and computing DWS for all requests must be amortized over throughput gains

Internal Use Cases:

  • RecSys serving: Multi-request types (rapid contextual rankings vs. bulk offline computations) on shared inference clusters
  • LLM APIs: Serving multiple client tiers (real-time chat vs. batch document processing) on shared GPUs
  • Search systems: Blending online ranking (few candidates, low latency) with offline deep learning reranking (many candidates, higher latency)
