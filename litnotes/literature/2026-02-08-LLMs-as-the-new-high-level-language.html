~fpereiro LLMs as the new high level language Hypothesis: LLM agents are the new high-level programming language (originally published here ) Following this hypothesis, what C did to assembler, what Java did to C, what Javascript/Python/Perl did to Java, now LLM agents are doing to all programming languages. What do I mean by LLM agents? I mean that the main development stack of a human will soon be: Multiple : a number of agents working in parallel. Autonomous : those agents only requiring feedback from the human every once in a while, but mostly work autonomously. How can we determine if the hypothesis is true? If a human developer can now build an order of magnitude more (10x) using multiple autonomous agents compared to what the human would be able to build without them, then the hypothesis is true. I’m not sure of it yet (as of January 2026) but I’m seriously considering it. For many that have been in the software business for a while, the mind reels with objections. Let’s address the easy ones first: 10x lines of code is not building 10x more, it’s just slop: the measure should be done on the actual functional output delivered, not the lines of code. If we go with the hypothesis, the “lines of code” are really the instructions to the LLM. LLMs are only for those who don’t know how to code: while there will be many new programmers thanks to LLMs, that doesn’t mean that experienced programmers won’t benefit from using LLM agents. Evidence shows that many experienced programmers are experiencing a lot more output thanks to LLMs. LLMs are for those who don’t want to think/work: if you are using LLMs to do more than you did before, you’ll have to think and work more, not less. It’s more demanding to manage a fleet of agents, and you’ll have to design way more (since you’re building x times what you were building before in the same amount of time). LLMs are going to make our coding skills rot: probably. But at work we are not usually concerned about our assembler, or our C chops rotting, if they exist at all. Most of us practice those chops in our free time, because we cannot defend the case that we’d be more productive working in assembler or C at work (for most types of software development). The code that LLMs make is much worse than what I can write: almost certainly; but the same could be said about your assembler, or your C code. LLM Agents as High-Level Programming Language

Core Idea: Multiple autonomous LLM agents working in parallel represent a new abstraction layer in programming, enabling developers to achieve 10x productivity gains similar to historical shifts from assembly → C → Java → Python.

Problem It Solves: Current programming languages still require developers to write detailed implementation code, limiting the scale and scope of what one person can build. The cognitive and time overhead of manual coding bottlenecks the translation from design to working software.

How It Works:

  1. Parallel Agent Orchestration: Developer deploys multiple specialized LLM agents that work concurrently on different aspects of the system (e.g., backend API, frontend UI, testing, documentation)
  2. Autonomous Execution with Human Checkpoints: Agents operate independently with minimal human intervention, only requesting feedback at key decision points or when encountering ambiguity
  3. Functional Output Measurement: Success is measured by delivered functionality rather than lines of code; developer instructions to LLMs become the new “source code”

Architecture:

    A -->|High-level instructions| C[Agent 2: Frontend]
    A -->|High-level instructions| D[Agent 3: Testing]

    B --> E[Functional System]
    C --> E
    D --> E

Where It’s Useful:

  • Full-stack application development where multiple components need simultaneous work
  • Rapid prototyping and MVP development where speed-to-market is critical
  • Refactoring and technical debt reduction across large codebases
  • Documentation and test generation alongside feature development

Limitations / Caveats:

  • Hypothesis still unproven as of January 2026; 10x productivity claim lacks rigorous empirical validation
  • Agent orchestration and management introduces new cognitive overhead (designing, coordinating, debugging multi-agent systems)
  • Code quality and architecture consistency concerns when agents work independently
  • Requires experienced developers who can effectively specify requirements and validate agent output
  • May lead to skill atrophy in manual coding, making developers dependent on agent availability

Internal Use Cases:

  • RecSys pipeline development: Deploy parallel agents for feature engineering, model training, evaluation metrics, and A/B test analysis
  • LLM fine-tuning workflows: One agent handles data preprocessing, another manages training jobs, a third runs evaluation benchmarks
  • Ranking model iteration: Agents simultaneously explore different architectures, hyperparameter spaces, and feature combinations
  • Production incident response: Multiple agents analyze logs, propose fixes, write tests, and update documentation in parallel
