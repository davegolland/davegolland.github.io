Decomposing Reasoning Efficiency in Large Language Models

Source: arXiv:cs.CL Published: 2026-02-10 Rank: #3 (score: 0.99, max_sim: 0.82) Reading time: ~20 min Tags: RecSys, LLM

Abstract

Large language models trained for reasoning trade off inference tokens against accuracy, yet standard evaluations report only final accuracy, obscuring where tokens are spent or wasted. We introduce a trace-optional framework that decomposes token efficiency into interpretable factors: completion under a fixed token budget (avoiding truncation), conditional correctness given completion, and verbosity (token usage). When benchmark metadata provides per-instance workload proxies, we further factor verbosity into two components: mean verbalization overhead (tokens per work unit) and a coupling coefficient capturing how overhead scales with task workload. When reasoning traces are available, we add deterministic trace-quality measures (grounding, repetition, prompt copying) to separate degenerate looping from verbose-but-engaged reasoning, avoiding human labeling and LLM judges. Evaluating 25 models on CogniLoad, we find that accuracy and token-efficiency rankings diverge (Spearman ), efficiency gaps are often driven by conditional correctness, and verbalization overhead varies by about 9 times (only weakly related to model scale). Our decomposition reveals distinct bottleneck profiles that suggest different efficiency interventions.

Key Techniques

  • Token budget completion analysis
  • conditional correctness factorization
  • verbosity decomposition with workload coupling
  • deterministic trace-quality measures

Trace-Optional Efficiency Decomposition

Core Idea: A systematic framework that decomposes LLM reasoning efficiency into interpretable factors (completion rate, conditional correctness, verbosity, and optional trace quality) to reveal where inference tokens are spent or wasted, enabling targeted optimization interventions.

Problem It Solves: Standard reasoning benchmarks report only final accuracy, obscuring critical efficiency bottlenecks—models may fail by running out of tokens, making reasoning errors, being unnecessarily verbose, or generating degenerate outputs. Without decomposition, practitioners cannot diagnose whether to optimize truncation handling, reasoning quality, or token overhead.

How It Works:

  1. Completion & Conditional Correctness: Measure completion rate under fixed token budgets (avoiding truncation) and accuracy conditional on completion, isolating budget failures from reasoning failures
  2. Verbosity Factorization: When workload proxies exist (e.g., proof steps, code lines), decompose total tokens into mean overhead per work unit and a coupling coefficient capturing how overhead scales with task difficulty
  3. Trace Quality Metrics (optional): For models with reasoning traces, compute deterministic metrics (grounding in problem context, repetition rate, prompt copying) to separate degenerate looping from engaged-but-verbose reasoning—no human annotation or LLM judges required
  4. Bottleneck Profiling: Combine factors to create efficiency profiles revealing whether models fail on budgets, correctness, or verbosity

Architecture:

Where It’s Useful:

  • Reasoning model development: Diagnosing whether o1/R1-style models waste tokens on verbosity vs. poor reasoning vs. budget mismanagement
  • Inference optimization: Identifying which models benefit from token budget increases vs. better stopping criteria vs. conciseness training
  • Benchmark design: Creating evaluations that measure not just “does it work” but “how efficiently does it work”
  • Model selection: Choosing between models with different accuracy/efficiency tradeoffs for production deployment

Limitations / Caveats:

  • Workload proxies (proof steps, code lines) must be available for fine-grained verbosity decomposition—not all benchmarks provide this metadata
  • Trace quality metrics assume reasoning is externalized in text; opaque models or models with implicit reasoning cannot be analyzed at this level
  • Coupling coefficient interpretation requires sufficient task workload variance; constant-difficulty benchmarks lose this signal
  • Framework focuses on single-pass reasoning; iterative refinement or multi-attempt strategies require extended analysis

Internal Use Cases:

  • LLM ranking systems: Decompose efficiency of candidate generation models to balance latency budgets—measure if verbosity comes from thoughtful feature consideration or repetitive looping
  • RecSys reasoning agents: Profile whether recommendation explanation models waste tokens on grounding failures vs. genuine complexity in multi-constraint optimization
  • Code generation pipelines: Separate verbosity from correctness in code-LLMs—identify if token overhead scales with code complexity (good coupling) or reflects degenerate patterns
  • Production model selection: Build efficiency profiles for model portfolios (e.g., Haiku vs. Sonnet vs. Opus) to route queries based on complexity workload proxies, maximizing throughput under SLA constraints
