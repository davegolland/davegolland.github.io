RedVisor: Reasoning-Aware Prompt Injection Defense via Zero-Copy KV Cache Reuse

Source: arXiv:cs.AI Published: 2026-02-02 Rank: #3 (score: 0.59, max_sim: 0.79) Reading time: ~0 min Tags: Early-Exit Code: https://arxiv.org/pdf/2602.01795v1

Abstract

Large Language Models (LLMs) are increasingly vulnerable to Prompt Injection (PI) attacks, where adversarial instructions hidden within retrieved contexts hijack the model’s execution flow. Current defenses typically face a critical trade-off: prevention-based fine-tuning often degrades general utility via the “alignment tax”, while detection-based filtering incurs prohibitive latency and memory costs. To bridge this gap, we propose RedVisor, a unified framework that synthesizes the explainability of detection systems with the seamless integration of prevention strategies. To the best of our knowledge, RedVisor is the first approach to leverage fine-grained reasoning paths to simultaneously detect attacks and guide the model’s safe response. We implement this via a lightweight, removable adapter positioned atop the frozen backbone. This adapter serves a dual function: it first generates an explainable analysis that precisely localizes the injection and articulates the threat, which then explicitly conditions the model to reject the malicious command. Uniquely, the adapter is active only during this reasoning phase and is effectively muted during the subsequent response generation. This architecture yields two distinct advantages: (1) it mathematically preserves the backbone’s original utility on benign inputs; and (2) it enables a novel KV Cache Reuse strategy, eliminating the redundant prefill computation inherent to decoupled pipelines. We further pioneer the integration of this defense into the vLLM serving engine with custom kernels. Experiments demonstrate that RedVisor outperforms state-of-the-art defenses in detection accuracy and throughput while incurring negligible utility loss.

Zero-Copy KV Cache Reuse Strategy

Core Idea: Eliminate redundant prefill computation in decoupled reasoning-response pipelines by reusing the Key-Value cache from the reasoning phase during response generation, avoiding duplicate forward passes over the same context.

Problem It Solves: Traditional detection-response pipelines require two separate forward passes—one for analysis and one for response generation—both processing identical context tokens. This doubles memory bandwidth and compute cost. KV cache reuse eliminates this redundancy by keeping attention cache entries valid across phases.

How It Works:

  1. Reasoning Phase: During adapter-guided reasoning, the backbone computes and caches Key-Value (KV) pairs for all input tokens (including context and query), storing them in the vLLM KV cache
  2. Cache Preservation: The cached KV pairs remain valid in GPU memory across the phase boundary because the input context does not change between reasoning and response phases
  3. Response Generation: During response generation, the backbone skips the prefill stage (which would recompute KV for input tokens) and directly begins decoding from the cached KV state, restoring only the output tokens generated during reasoning to the cache and reusing all context KVs

Where It’s Useful:

  • Serving inference pipelines where reasoning and response generation are sequential (detection-then-respond patterns)
  • vLLM and other KV-cache-aware inference engines with custom kernel support
  • High-throughput LLM serving where latency and memory bandwidth are bottlenecks
  • Any multi-stage decoding pipeline where intermediate states are cached

Limitations / Caveats:

  • Requires careful synchronization between phase boundaries to ensure cache coherence; kernel-level integration is necessary for correctness
  • Benefit is marginal for very short contexts where prefill is already negligible relative to decoding cost
  • Compatibility depends on the inference engine’s support for mid-generation cache manipulation (not all inference stacks support this)

Internal Use Cases:

  • Multi-stage ranking pipelines: Cache context features during initial ranking pass, reuse during scoring/calibration stage in two-stage ranking systems
  • Streaming RAG systems: Reuse cached embeddings and KV states across retrievals and re-rankings without recomputing context prefills
  • Interactive recommendation refinement: Store user profile KV cache during initial filtering, reuse during personalization ranking to avoid recomputation

Integration with vLLM Serving Engine

Core Idea: Implement RedVisor’s reasoning adapter and KV cache reuse as custom kernels within the vLLM serving engine, enabling efficient, batched inference with multi-phase request handling.

Problem It Solves: A naive implementation of reasoning-aware defense as a post-hoc wrapper incurs significant overhead due to redundant context loading, unbatched adapter execution, and cache format mismatches. Direct vLLM integration enables (1) seamless request batching across multiple phases, (2) zero-copy cache transfers, and (3) hardware-optimized kernel execution.

How It Works:

  1. Request Queuing: Incoming requests are batched and routed to the reasoning phase; the vLLM scheduler queues adapter inference on the backbone, with requests grouped by context length for efficient computation
  2. Adapter Execution: Custom vLLM kernels execute the adapter module within the same GPU memory space as the backbone, preserving KV cache locality
  3. Phase Transition: After reasoning, the same batched requests are transitioned to response generation; the scheduler maintains KV cache ownership and skips prefill, directly dispatching decoding kernels

Where It’s Useful:

  • Production LLM serving infrastructure (vLLM, TensorRT-LLM, SGLang, etc.)
  • Multi-tenant serving scenarios where request batching across security phases improves hardware utilization
  • Deployments requiring transparent integration into existing inference pipelines

Limitations / Caveats:

  • Requires custom kernel development and deep knowledge of the serving engine’s internals; not portable across different inference frameworks without additional engineering
  • Debugging and profiling multi-phase batched execution is complex and less transparent than simple sequential execution
  • Vendor lock-in to vLLM ecosystem; changes to vLLM’s cache management or scheduling may require adapter maintenance

Internal Use Cases:

  • Batch recommendation ranking: Integrate RedVisor into vLLM-based ranking services to defend batch inference on retrieved candidate sets while maintaining high throughput
  • Collaborative filtering with LLM explanations: Protect explanation generation pipeline in vLLM-served personalized recommendation systems from injection attacks
  • Large-scale content moderation: Implement reasoning-aware moderation detection at serving scale using vLLM’s batching capabilities

Analysis: RedVisor in Context of Prior Work

RedVisor addresses a fundamental tension in LLM defense that becomes especially critical when viewed through the lens of LLM Inference Performance. The paper’s key innovation—the “zero-copy KV cache reuse” strategy—directly tackles the decode-phase bottleneck identified in the Team Mental Model. Traditional detection-based defenses create two separate inference passes (one for reasoning/detection, one for response generation), which means redundant prefill computation and multiple KV cache instantiations. RedVisor’s adapter-based approach that is “active only during the reasoning phase and muted during response generation” cleverly exploits the inference pipeline structure: by reusing the KV cache from the detection phase, it converts what would be two full prefill operations into a single prefill with a lightweight reasoning pass. This is a concrete instantiation of the principle that “everything works by shrinking one of those terms”—in this case, the attention-on-context and KV-cache-reads terms for the second pass.

The connection to Early-Exit Language Models is more subtle but important. Both papers recognize that not all computation is equally necessary across the inference pipeline. Early-exit systems ask whether we can terminate early on “easy” inputs; RedVisor asks whether we can selectively activate/deactivate reasoning capacity based on threat detection needs. The adapter mechanism mirrors the early-exit pattern of conditional computation: maintaining a frozen backbone while adding lightweight, removable logic that operates conditionally. However, RedVisor’s motivation differs—it’s not about confidence thresholds on normal inputs, but about injecting reasoning paths that explicitly guide safe behavior when adversarial content is detected.

Finally, RedVisor’s approach of decomposing the defense into fine-grained reasoning that localizes attacks echoes the principles in Recursive Language Models, which emphasize that LLMs can “programmatically examine, decompose, and recursively call itself.” While RLMs apply this to handle context-length constraints, RedVisor applies recursive reasoning to handle adversarial constraints—the model must examine the prompt structure, identify injected instructions, and recursively condition its response generation on that analysis. Both papers leverage the insight that LLMs’ reasoning capabilities can be structured and controlled through careful prompt/architecture design. RedVisor’s novel contribution is demonstrating that this structured reasoning can simultaneously serve defense (detecting attacks) and efficiency (avoiding redundant computation through KV reuse) without incurring the “alignment tax” of prevention-based fine-tuning.
