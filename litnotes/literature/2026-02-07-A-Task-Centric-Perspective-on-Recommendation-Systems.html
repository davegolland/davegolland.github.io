A Task-Centric Perspective on Recommendation Systems

Source: S2:RecSys Published: 2025 Rank: #2 (score: 0.96, max_sim: 0.84) Reading time: ~20 min Tags: RecSys, LLM, Search

Abstract

Many studies in recommender systems (RecSys) adopt a general problem definition, i.e., to recommend preferred items to users based on past interactions. Such abstraction often lacks the domain-specific nuances necessary for practical deployment. However, models are frequently evaluated using datasets collected from online recommender platforms, which inherently reflect domain or task specificities. In this paper, we analyze RecSys task formulations, emphasizing key components such as input-output structures, temporal dynamics, and candidate item selection. All these factors directly impact offline evaluation. We further examine the complexities of user-item interactions, including decision-making costs, multi-step engagements, and unobservable interactions, which may influence model design. Additionally, we explore the balance between task specificity and model generalizability, highlighting how well-defined task formulations serve as the foundation for robust evaluation and effective solution development. By clarifying task definitions and their implications, this work provides a structured perspective on RecSys research. The goal is to help researchers better navigate the field, particularly in understanding specificities of the RecSys tasks and ensuring fair and meaningful evaluations.

Task-Centric RecSys Formulation Framework

Core Idea: A structured analytical framework that decomposes recommender system problems into explicit task components (input-output structure, temporal dynamics, candidate selection, interaction complexity) to enable fair evaluation and domain-appropriate model design.

Problem It Solves: Addresses the mismatch between abstract “recommend items to users” problem definitions and the domain-specific realities of deployed systems, which leads to unfair benchmarking, inappropriate model choices, and evaluation metrics that don’t reflect real-world performance.

How It Works:

  1. Input-Output Structure Analysis - Define what data is available at inference (user history, context, session state) and what format the output takes (ranked list, single item, explanation)
  2. Temporal Dynamics Specification - Characterize whether predictions are next-item, next-basket, long-term preference, or incorporate time-varying factors
  3. Candidate Selection Strategy - Explicitly define the candidate pool (full catalog, pre-filtered set, dynamic generation) and how it’s constructed
  4. Interaction Complexity Mapping - Model multi-step engagement patterns, decision costs, and unobservable user actions that standard metrics miss

Where It’s Useful:

  • Cross-domain RecSys research where comparing methods across different platforms (e.g., e-commerce vs. streaming vs. social feeds)
  • Designing offline evaluation protocols that better predict online performance
  • Matching model capabilities to task requirements (e.g., sequential models for session-based tasks, multi-objective models for platforms with multiple engagement types)
  • Identifying when benchmark results are not comparable due to task differences

Limitations / Caveats:

  • Framework is analytical/conceptual rather than algorithmic - doesn’t provide direct model improvements
  • Requires domain expertise to properly characterize task components
  • May increase complexity of benchmark design and reporting
  • Doesn’t resolve fundamental online/offline evaluation gaps, only clarifies them

Internal Use Cases:

  • Multi-task learning design: Use task formulation analysis to identify which tasks share structure and can benefit from shared representations vs. which require task-specific towers
  • Evaluation protocol design: Map candidate selection strategies to appropriate negative sampling schemes and ranking metrics
  • Model-task matching: Determine when autoregressive LLMs are appropriate (generative item creation tasks) vs. retrieval models (fixed catalog ranking)
  • Dataset documentation: Standardize task specification in benchmark documentation to enable fair comparison and appropriate method selection
