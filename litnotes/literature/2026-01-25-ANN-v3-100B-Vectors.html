ANN v3: 200ms p99 query latency over 100 billion vectors

ANN v3: 200ms p99 query latency over 100 billion vectors

Summary

Turbopuffer rebuilt their vector search to handle 100 billion vectors at 200ms p99 latency and >1k QPS. This is a significant engineering achievement—100B vectors at 1024 dimensions in f16 is 200 TiB of data. The key insight is that vector search is bandwidth-bound, not compute-bound, so optimization must focus on memory hierarchy utilization rather than faster distance calculations.

The Core Insight: Arithmetic Intensity

Vector search has low arithmetic intensity. In a dot product, each data element is used exactly once—you multiply d[i] * q[i] and move on. Compare this to matrix multiplication where each element participates in N operations.

Low arithmetic intensity means: you’re limited by how fast you can feed data to the CPU, not how fast the CPU can process it.

This is why naive optimizations (faster SIMD instructions, GPU kernels) don’t help much—you’re already waiting on memory. The solution is to either:

  1. Reduce how much data you need to load (approximation)
  2. Keep frequently-accessed data in faster memory tiers (caching hierarchy)

Turbopuffer does both.

Technique 1: Hierarchical Clustering (SPFresh)

Instead of searching all 100B vectors, cluster them hierarchically:

Root centroid
├── Centroid C1
│   ├── Data vectors v1, v2, ...
│   └── ...
├── Centroid C2
│   └── ...
└── Centroid C3
    └── ...

At query time:

  1. Compare query to root centroids → identify promising subtrees
  2. Descend into those subtrees, compare to child centroids
  3. Only scan data vectors in the final selected clusters

With a 100x branching factor at each level, a 100B vector index has ~4 levels. You scan maybe 500 clusters × 100 vectors = 50k vectors instead of 100B. That’s a 2-million-fold reduction in data touched.

Why 100x Branching?

This is clever: the branching factor roughly matches the DRAM-to-SSD size ratio (10-50x). If all data vectors fit on SSD, all centroid vectors fit in DRAM. Upper tree levels stay hot in fast memory; data vectors come from slower storage only when needed.

Spatial and Temporal Locality

Clustering provides:

  • Spatial locality: Vectors close in embedding space are stored contiguously → efficient sequential reads
  • Temporal locality: Upper tree levels accessed on every query → stay resident in cache

Technique 2: Binary Quantization (RaBitQ)

Even with clustering, you still need to scan ~50k vectors per query at ~2KB each = 100MB of bandwidth per tree level. At SSD speeds (~10 GB/s), that limits you to ~100 QPS.

Solution: compress the vectors. Binary quantization converts f16 (2 bytes/dim) to 1 bit/dim—a 16-32x compression.

[0.94, -0.01, 0.39, -0.72, ...] → [1, 0, 1, 0, ...]

Now those 50k vectors fit in ~6MB instead of 100MB. They can live higher in the memory hierarchy.

Why Doesn’t This Kill Accuracy?

RaBitQ exploits concentration of measure in high dimensions. In high-D space, vector components become more uniformly distributed, so quantization errors spread evenly rather than concentrating. The result: 1-bit quantization with surprisingly high recall.

The system stores both quantized vectors (for fast approximate search) and full-precision vectors (for final re-ranking of top candidates). Approximation + refinement.

The Memory Hierarchy

Tier              Size        Bandwidth
─────────────────────────────────────────
CPU Registers     <1KB        >10 TB/s
L1/L2/L3 Cache    KB-MB       1-10s TB/s
DRAM              GB-TB       100-500 GB/s
NVMe SSD          1-10s TB    1-30 GB/s
Object Storage    PB-EB       1-10 GB/s

The optimization goal: don’t let any single tier become the bottleneck. If you max out SSD bandwidth while DRAM sits idle, you’ve left performance on the table.

Turbopuffer’s architecture:

  • Stateless query tier over S3
  • NVMe as a cache layer
  • Quantized vectors sized to fit in DRAM
  • Centroids sized to fit in L3

Practical Takeaways

  1. Think about bandwidth, not compute for vector workloads
  2. Hierarchical clustering reduces search space by orders of magnitude
  3. Binary quantization (especially RaBitQ) is surprisingly effective
  4. Match data structure sizes to memory tier sizes for balanced utilization
  5. The “approximation + refinement” pattern applies broadly
