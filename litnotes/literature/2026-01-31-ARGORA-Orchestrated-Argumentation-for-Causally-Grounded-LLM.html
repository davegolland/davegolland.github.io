ARGORA: Orchestrated Argumentation for Causally Grounded LLM Reasoning and Decision Making

Source: arXiv:cs.AI Published: 2026-01-29 Rank: #7 (score: 0.99, max_sim: 0.83) Reading time: ~15 min Tags: LLM, RecSys

Abstract

Existing multi-expert LLM systems gather diverse perspectives but combine them through simple aggregation, obscuring which arguments drove the final decision. We introduce ARGORA, a framework that organizes multi-expert discussions into explicit argumentation graphs showing which arguments support or attack each other. By casting these graphs as causal models, ARGORA can systematically remove individual arguments and recompute outcomes, identifying which reasoning chains were necessary and whether decisions would change under targeted modifications. We further introduce a correction mechanism that aligns internal reasoning with external judgments when they disagree. Across diverse benchmarks and an open-ended use case, ARGORA achieves competitive accuracy and demonstrates corrective behavior: when experts initially disagree, the framework resolves disputes toward correct answers more often than it introduces new errors, while providing causal diagnostics of decisive arguments.

Key Techniques

  • Argumentation graphs
  • Causal intervention analysis
  • Argument-based correction mechanism
  • Multi-expert deliberation framework

Causal Argument Removal and Sensitivity Analysis

Core Idea: Cast the argumentation graph as a causal model, then systematically remove individual arguments or argument clusters and recompute outcomes to identify which arguments are causally necessary for the final decision.

Problem It Solves: Even with explicit argumentation structure, determining which arguments actually influenced the outcome (vs. being present but irrelevant) requires a principled intervention mechanism. Simple ablation is not principled for argumentation; causal removal ensures we’re measuring true causal necessity.

How It Works:

  1. Given a constructed argumentation graph G with arguments {A₁, A₂, …, Aₘ} and edges representing support/attack relationships
  2. For each argument Aᵢ (or subset of arguments S ⊆ {A₁, …, Aₘ}):
    • Remove Aᵢ (and its incoming/outgoing edges) from G, creating G₋ᵢ
    • Recompute the final decision/outcome using the modified graph G₋ᵢ (e.g., via aggregation over remaining arguments)
    • Compare original outcome with outcome on G₋ᵢ
  3. If outcome changes, Aᵢ is causally necessary; if it remains unchanged, Aᵢ is causally inert for this decision
  4. Rank arguments by impact: those whose removal most drastically changes the outcome are most causal
  5. Identify decisive argument chains: paths in the graph from initial arguments to the final decision that would break if any single argument were removed

Where It’s Useful:

  • Debugging LLM multi-expert systems: identify which expert contributions actually matter
  • Regulatory compliance: demonstrate to auditors which reasoning was necessary for a decision
  • Interactive decision refinement: highlight which arguments should be revisited if the outcome needs to change

Limitations / Caveats:

  • Assumes clean removal is possible; in practice, arguments may have implicit dependencies not captured in the explicit graph
  • Requires a deterministic or well-defined way to recompute outcomes after removal; unclear outcomes may persist under different subsets
  • Computational cost is O(m) or higher (one recomputation per argument); for large graphs this can be expensive
  • Causal reasoning is only as good as the graph structure—missing or mis-labeled edges will hide true causal relationships

Internal Use Cases:

  • Ranking system debugging: remove candidate ranking factors one at a time to see which truly impact final ranking
  • A/B testing alternatives: “if we remove the engagement factor from the ranking, does top-k change?”

Correction Mechanism for Disagreement Resolution

Core Idea: When expert opinions disagree with external ground truth (e.g., held-out labels), apply a correction process that adjusts internal reasoning/argument weights to align with correct answers, then re-aggregate to resolve disputes toward the right conclusion.

Problem It Solves: Multi-expert systems may reach incorrect consensus if all experts are biased in the same direction, or if they agree on premises that are actually false. Simple aggregation cannot self-correct. This mechanism detects when expert consensus diverges from ground truth and applies a targeted correction.

How It Works:

  1. Run the multi-expert argumentation system (with graph construction and aggregation) on a set of instances with known ground truth labels Y_true
  2. Identify discordance cases: instances where the system’s conclusion disagrees with Y_true
  3. For discordance cases, perform causal removal analysis (from technique 2) to identify which arguments/experts contributed most to the wrong conclusion
  4. Apply a correction weight or re-weighting to those arguments/experts based on their role in errors:
    • Arguments that consistently appear in error chains are down-weighted or adjusted
    • Arguments that support the correct answer but were overridden are up-weighted
  5. Re-aggregate the argumentation graph with corrected weights to produce a revised decision
  6. (Optional) Fine-tune or prompt-engineer the expert LLMs to reduce bias toward incorrect premises

Architecture:

flowchart LR
    A["Multi-Expert<br/>Argumentation System"] --> B{Prediction =<br/>Ground Truth?}
    B -->|Yes| C["Accept Decision"]
    B -->|No| D["Causal Removal<br/>Analysis on Error"]
    D --> E["Identify Contributing<br/>Arguments"]
    E --> F["Adjust Argument<br/>Weights/Corrections"]
    F --> G["Re-aggregate with<br/>Corrected Weights"]
    G --> H["Revised Decision"]

Where It’s Useful:

  • Learning from human feedback: when the system disagrees with a human expert, automatically identify which parts of the reasoning were wrong
  • Active learning: prioritize which expert arguments need improvement based on where corrections are most frequent
  • Alignment to external evaluators: ensure internal LLM reasoning aligns with external criteria (compliance, domain expertise, fairness)

Limitations / Caveats:

  • Requires ground truth labels to detect and correct discordance; unsuitable for truly open-ended problems without clear correctness
  • Correction mechanism may overfit to the training set of ground truth labels, reducing generalization
  • Assumes causal removal accurately identifies error contributors; noisy graphs may misdirect corrections
  • Introduces additional LLM calls (for causal analysis), increasing computational cost

Internal Use Cases:

  • Recommendation system calibration: when recommendations disagree with user feedback (“I didn’t like that”), identify which recommendation factors were wrong and down-weight them
  • Ranking correction: if ranking results diverge from ground-truth relevance (e.g., from human judges), apply correction to the argument factors driving the ranking
