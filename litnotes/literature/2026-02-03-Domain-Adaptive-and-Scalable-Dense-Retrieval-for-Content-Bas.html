Domain-Adaptive and Scalable Dense Retrieval for Content-Based Recommendation

Domain-Adaptive and Scalable Dense Retrieval for Content-Based Recommendation

Source: arXiv:cs.IR Published: 2026-01-31 Rank: #2 (score: 0.95, max_sim: 0.85) Reading time: ~0 min Tags: Uncategorized Code: https://arxiv.org/pdf/2602.00899v1

Abstract

E-commerce recommendation and search commonly rely on sparse keyword matching (e.g., BM25), which breaks down under vocabulary mismatch when user intent has limited lexical overlap with product metadata. We cast content-based recommendation as recommendation-as-retrieval: given a natural-language intent signal (a query or review), retrieve the top-K most relevant items from a large catalog via semantic similarity. We present a scalable dense retrieval system based on a two-tower bi-encoder, fine-tuned on the Amazon Reviews 2023 (Fashion) subset using supervised contrastive learning with Multiple Negatives Ranking Loss. We construct training pairs from review text (as a query proxy) and item metadata (as the positive document) and fine-tune on 50,000 sampled interactions with a maximum sequence length of 500 tokens. For efficient serving, we combine FAISS HNSW indexing with an ONNX Runtime inference pipeline using INT8 dynamic quantization. On a review-to-title benchmark over 826,402 catalog items, our approach improves Recall@10 from 0.26 (BM25) to 0.66, while meeting practical latency and model-size constraints: 6.1 ms median CPU inference latency (batch size 1) and a 4x reduction in model size. Overall, we provide an end-to-end, reproducible blueprint for taking domain-adapted dense retrieval from offline training to CPU-efficient serving at catalog scale.

Key Techniques

  • Dense retrieval with bi-encoders
  • Supervised contrastive learning with Multiple Negatives Ranking Loss
  • FAISS HNSW indexing with INT8 dynamic quantization

FAISS HNSW Indexing for Approximate Nearest Neighbor Search

Core Idea: Use Hierarchical Navigable Small World (HNSW) graphs indexed via FAISS to enable sub-linear retrieval latency across 826,402 catalog items by organizing embeddings into a hierarchical graph structure supporting efficient approximate nearest-neighbor queries.

Problem It Solves:

  • Exact nearest-neighbor search in high dimensions requires O(n) similarity computations per query, infeasible at catalog scale
  • Brute-force retrieval bottlenecks CPU serving latency; HNSW trades negligible recall loss for 100-1000x speedup
  • FAISS provides optimized, production-ready indexing without custom implementation overhead

How It Works:

  1. Offline Indexing: Encode all 826,402 items via document encoder E_d; insert embeddings into FAISS HNSW index with M (max connections per node) and ef_construction (expansion factor) tuned for latency-recall tradeoff
  2. Query Execution: At inference time, encode incoming query; HNSW performs hierarchical graph traversal starting from randomly selected entry point, greedily navigating to nearest neighbors at each level
  3. Top-K Retrieval: Return top-K items (K=10 in the paper’s benchmark) with highest embedding similarity to query, found via HNSW graph traversal in ~O(log n) expected time

Architecture:

flowchart TD
    A["Query Embedding"] --> B["HNSW Entry Point"]
    B --> C["Graph Traversal (Level 0..L)"]
    C --> D["Candidate Pool Expansion"]
    D --> E["Sort by Similarity"]
    E --> F["Top-K Results"]

Where It’s Useful:

  • Large-scale dense retrieval (100K+ items) where latency is critical
  • Production recommendation systems requiring sub-10ms retrieval
  • Any semantic search application (search engines, QA, document retrieval)

Limitations / Caveats:

  • Approximate search: achieves 0.66 Recall@10 vs. 1.0 for exact search (though sparse BM25 baseline only reaches 0.26)
  • Index construction is sequential and memory-intensive; rebuilding on catalog updates requires offline reindexing
  • HNSW hyperparameters (M, ef_construction, ef_search) must be tuned per dataset; poor tuning degrades both latency and recall
  • No learned component; index structure is independent of downstream ranking objectives

Internal Use Cases:

  • RecSys candidate retrieval: Use HNSW to retrieve 100-1000 candidates efficiently, feed to neural ranker for final ranking
  • LLM-based recommendation: Retrieve semantically similar items before prompting LLM for personalized ranking
  • A/B testing infrastructure: HNSW enables fast baseline serving for dense retrieval experiments vs. sparse methods

ONNX Runtime INT8 Dynamic Quantization for CPU-Efficient Serving

Core Idea: Convert floating-point encoder weights and activations to 8-bit integers at inference time using ONNX Runtime’s dynamic quantization, achieving 4x model size reduction and CPU-compatible latency (6.1 ms per query, batch size 1) with minimal recall loss.

Problem It Solves:

  • Full-precision dense retrievers (typically 100-500 MB) exceed memory budgets on CPU-only serving infrastructure
  • GPU serving incurs latency SLO violations and operational cost for real-time retrieval; CPU inference is preferred when feasible
  • Static quantization requires calibration on representative data; dynamic quantization avoids this offline step

How It Works:

  1. Model Export: Serialize trained bi-encoder (query and document encoders) to ONNX format, preserving weights in FP32
  2. Dynamic Quantization: At load time, ONNX Runtime converts FP32 weights → INT8 and activations to INT8 during forward pass; quantization parameters (scale, zero-point) computed per-tensor based on observed activation ranges
  3. Inference Execution: Run quantized forward pass on CPU; ONNX Runtime automatically dequantizes outputs back to FP32 for downstream similarity computation
  4. Size Reduction: INT8 weights are 4x smaller (1 byte vs. 4 bytes per parameter), reducing memory footprint from ~200-400 MB → ~50-100 MB

Architecture:

flowchart LR
    A["Input Text"] --> B["ONNX Tokenizer"]
    B --> C["INT8 Encoder Weights"]
    C --> D["Quantized MatMul"]
    D --> E["INT8 Activation"]
    E --> F["Dequantize to FP32"]
    F --> G["Query/Doc Embedding"]

Where It’s Useful:

  • Edge and on-device inference (mobile, IoT) where model size is critical
  • CPU-only serving environments (cloud VMs without GPU, cost-sensitive deployments)
  • Low-latency serving where full batch processing is infeasible and per-sample latency < 10ms is required

Limitations / Caveats:

  • Quantization noise degrades ranking quality; recall typically drops 1-5% depending on model and quantization scheme
  • Dynamic quantization adds per-inference overhead for activation range computation; not suitable for GPU inference (GPU static quantization is faster)
  • INT8 compute benefits only realized on CPUs with specialized instructions (AVX2, AVX-512); older CPUs may show no latency gain
  • Tradeoff: achieves 4x size reduction but absolute latency (6.1 ms) still exceeds streaming latency budgets for real-time personalization

Internal Use Cases:

  • RecSys production serving: Deploy quantized encoders on CPU-only nodes for candidate retrieval in two-stage pipelines
  • LLM inference: Use quantized dense retrievers to fetch context documents before LLM generation, reducing total latency
  • Cost optimization: Replace GPU serving with quantized CPU inference for batch retrieval jobs (e.g., pre-computing embeddings for 1M+ new items)

Analysis: Domain-Adaptive Dense Retrieval and the Evolving Landscape of Information Retrieval

This paper operationalizes Transformer-based Dense Retrieval in a practical e-commerce setting, demonstrating how the theoretical advantages of semantic similarity-based ranking translate to real-world recommendation systems. While the matched notes describe dual-encoder BERT architectures encoding queries and documents individually for similarity comparison, this paper grounds that concept in a concrete pipeline: it fine-tunes a two-tower bi-encoder on Amazon product-review pairs, achieving a Recall@10 of 0.66 versus BM25’s 0.26. Crucially, it addresses a gap left implicit in the dense retrieval literature—the scalability and efficiency requirements of serving these models at catalog scale (826K+ items). By integrating FAISS HNSW indexing with INT8 dynamic quantization and ONNX Runtime, the paper extends dense retrieval from a research concept to a deployable system, reducing model size by 4x while maintaining 6.1ms CPU inference latency. This bridges the abstraction gap between the “how it works” description in the matched notes and the “how it ships” reality of production systems.

The paper’s use of supervised contrastive learning with Multiple Negatives Ranking Loss implicitly aligns with the intent-aware spirit of Instruction-Based Dense Retrieval with LLM Encoders, though through a different mechanism. Rather than explicit natural-language instructions, this work uses review text as an implicit signal of user intent, treating reviews as query proxies paired with item metadata. This is a domain adaptation technique: it doesn’t require external instructions but instead leverages the existing signal structure within reviews themselves. The distinction is subtle but important—while instruction-based approaches ask models to follow explicit commands, this paper shows that domain-specific signals (review-to-product matching) can serve as equally powerful supervisory signals for fine-tuning. Both approaches attempt to inject task-specific understanding into dense retrievers, but this paper does so through data curation and loss formulation rather than through instruction augmentation.

Finally, this work extends the LLM Empowered Agents for Recommendation and Search survey implicitly by demonstrating that dense retrieval can work effectively without large language models. Rather than deploying LLM agents as described in the matched notes—where agents compose multi-step reasoning over retrieval results—this paper shows that a smaller, fine-tuned transformer bi-encoder can be more practical for e-commerce at scale. It represents a pragmatic alternative: domain adaptation via supervised contrastive learning often yields better latency-accuracy tradeoffs than LLM-based approaches for well-defined retrieval tasks with bounded catalogs. This perspective challenges the implicit assumption in LLM-agent literature that larger, more sophisticated models are always necessary, showing instead that task-specific fine-tuning on domain data can be more efficient.
