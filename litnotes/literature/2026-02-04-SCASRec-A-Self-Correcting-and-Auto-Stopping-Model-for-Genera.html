SCASRec: A Self-Correcting and Auto-Stopping Model for Generative Route List Recommendation

Source: arXiv:cs.IR Published: 2026-02-03 Rank: #8 (score: 0.98, max_sim: 0.83) Reading time: ~20 min Tags: RecSys, LLM, RL Code: https://arxiv.org/pdf/2602.03324v1

Abstract

Route recommendation systems commonly adopt a multi-stage pipeline involving fine-ranking and re-ranking to produce high-quality ordered recommendations. However, this paradigm faces three critical limitations. First, there is a misalignment between offline training objectives and online metrics. Offline gains do not necessarily translate to online improvements. Actual performance must be validated through A/B testing, which may potentially compromise the user experience. Second, redundancy elimination relies on rigid, handcrafted rules that lack adaptability to the high variance in user intent and the unstructured complexity of real-world scenarios. Third, the strict separation between fine-ranking and re-ranking stages leads to sub-optimal performance. Since each module is optimized in isolation, the fine-ranking stage remains oblivious to the list-level objectives (e.g., diversity) targeted by the re-ranker, thereby preventing the system from achieving a jointly optimized global optimum. To overcome these intertwined challenges, we propose \textbf{SCASRec} (\textbf{S}elf-\textbf{C}orrecting and \textbf{A}uto-\textbf{S}topping \textbf{Rec}ommendation), a unified generative framework that integrates ranking and redundancy elimination into a single end-to-end process. SCASRec introduces a stepwise corrective reward (SCR) to guide list-wise refinement by focusing on hard samples, and employs a learnable End-of-Recommendation (EOR) token to terminate generation adaptively when no further improvement is expected. Experiments on two large-scale, open-sourced route recommendation datasets demonstrate that SCASRec establishes an SOTA in offline and online settings. SCASRec has been fully deployed in a real-world navigation app, demonstrating its effectiveness.

Key Techniques

  • Stepwise corrective reward (SCR)
  • learnable End-of-Recommendation (EOR) token
  • unified generative framework

End-of-Recommendation (EOR) Token with Adaptive Stopping

Core Idea: A learnable special token that signals when to terminate generation, allowing the model to dynamically decide list length based on learned quality cues rather than fixed output constraints.

Problem It Solves: Traditional ranking systems output a fixed number of candidates, including low-quality or redundant items that degrade list quality. Rigid cutoff rules cannot adapt to variance in user intent and recommendation difficulty across different scenarios.

How It Works:

  1. Introduce a special learnable token with embeddings
  2. During generation at step , compute logits for all candidate routes plus the EOR token:
  3. Apply softmax to all candidates + EOR token and sample next action (either select a route or generate EOR)
  4. If EOR is sampled, terminate list generation immediately; otherwise continue with selected route
  5. Train EOR token via policy gradient or reinforcement learning to predict when marginal list quality gain falls below a learned threshold

Architecture:

flowchart LR
    A["Hidden State h_t"] --> B["Score All Routes + EOR"]
    B --> C["Softmax over Routes ∪ {EOR}"]
    C --> D{Sample Action}
    D -->|Select Route| E["Append Route to List"]
    D -->|EOR Token| F["Terminate Generation"]
    E --> G["Update h_{t+1}"]
    G --> C
    F --> H["Output Final List"]

Where It’s Useful:

  • Sequential recommendation where list length should vary by user/context
  • Route recommendation with variable scenario complexity
  • Any ranking task where quality-per-item degrades and optimal list length is data-dependent
  • Real-world systems where showing too many low-value options harms UX

Limitations / Caveats:

  • EOR token must be trained jointly with ranking—poor initialization can lead to premature or excessive termination
  • Requires reward signal that clearly indicates when to stop (avoids ambiguity)
  • May require curriculum learning or careful initialization to prevent collapse to always-stopping or never-stopping behavior

Internal Use Cases:

  • Addresses User Feedback Alignment by allowing the model to learn when user exploration should stop based on learned satisfaction signals
  • Improves the Two-Stage LLM Sequential Recommendation paradigm by making next-item stopping decisions part of a unified generative process

Unified Generative Ranking Framework (End-to-End Fine-Ranking + Re-Ranking Integration)

Core Idea: A single generative model that jointly performs ranking and redundancy elimination in one pass, replacing the traditional multi-stage pipeline with an integrated end-to-end process that optimizes for both relevance and list-level objectives simultaneously.

Problem It Solves: Classical two-stage pipelines (fine-ranking → re-ranking) suffer from: (1) stage separation causing each to ignore downstream objectives, (2) misalignment between offline metrics and online A/B test results, and (3) rigid handcrafted redundancy rules that lack adaptability.

How It Works:

  1. Encode user context, query, and candidate routes into a shared representation space
  2. Use a transformer-based decoder that autoregressively generates the recommendation list token-by-token
  3. At each step , condition on the current partial list and hidden state to score remaining candidates
  4. Integrate Stepwise Corrective Reward (SCR) into the loss to backpropagate list-level objectives (diversity, redundancy) into ranking decisions
  5. Use End-of-Recommendation (EOR) token to determine when to stop generation
  6. Train end-to-end with RL or policy gradient methods that directly optimize online metrics (rather than offline ranking metrics)

Architecture:

flowchart LR
    A["User Context + Query"] --> B["Shared Encoder"]
    B --> C["Autoregressive Decoder"]
    C --> D["Score Candidates"]
    D --> E{"Hard Sample?"}
    E -->|Yes| F["Apply SCR Loss Upweighting"]
    E -->|No| G["Standard Loss"]
    F --> H["Generate Next Route or EOR"]
    G --> H
    H --> I{EOR Token?}
    I -->|Yes| J["Stop"]
    I -->|No| K["Append Route, Update h_t"]
    K --> D
    J --> L["Final Ranked List"]

Where It’s Useful:

  • Large-scale recommendation systems (navigation, e-commerce, content recommendation) where multi-stage pipelines introduce latency and suboptimality
  • Scenarios requiring joint optimization of relevance + diversity + coverage
  • Real-world deployed systems where A/B testing cost is high and offline-online alignment is critical
  • Route recommendation and navigation where redundancy matters (avoid repeated turns/highways)

Limitations / Caveats:

  • Requires joint training signal that includes both ranking and list-level objectives—harder to train than modular pipelines
  • Generative decoding is slower than indexing-based retrieval; inference latency may increase
  • End-to-end learning with RL introduces training complexity and potential instability
  • Requires careful design of reward functions to avoid gaming or unexpected behavior

Internal Use Cases:

  • Directly unifies insights from LLM-Driven Post-hoc Re-ranking and Explanation (re-ranking as a learned stage) with Two-Stage LLM Sequential Recommendation (sequential generation)
  • Applies User Feedback Alignment by making the stopping decision (when to end list) part of the core generative objective, enabling adaptive exploration-exploitation tradeoffs
