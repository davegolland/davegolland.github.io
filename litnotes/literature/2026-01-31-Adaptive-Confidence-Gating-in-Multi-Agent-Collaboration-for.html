Adaptive Confidence Gating in Multi-Agent Collaboration for Efficient and Optimized Code Generation

Source: arXiv:cs.AI Published: 2026-01-29 Rank: #12 (score: 0.97, max_sim: 0.84) Reading time: ~20 min Tags: LLM

Abstract

While Large Language Models (LLMs) have catalyzed breakthroughs in automated code generation, Small Language Models (SLMs) often encounter reasoning bottlenecks and failure loops when addressing complex logical requirements. To overcome these challenges, we propose DebateCoder, a multi-agent collaborative framework designed to improve the reasoning ability of SLMs (e.g., Pangu-1B) in resource-constrained environments. DebateCoder uses a structured role-playing protocol with three agents: User Agent (A_UA), Technical Agent (A_TA), and Quality Assurance Agent (A_QA). It also includes an Adaptive Confidence Gating mechanism with a 95% threshold to balance accuracy and inference efficiency. In addition, we introduce a multi-turn deliberation module and a reviewer-guided analytical debugging loop for orthogonal pre-generation debate and post-generation refinement. Experiments on HumanEval and MBPP show that DebateCoder achieves 70.12% Pass@1 on HumanEval, outperforming MapCoder while reducing API overhead by about 35%. These results indicate that collaborative protocols can mitigate limitations of small-parameter models and provide a scalable, efficient approach to high-quality automated software engineering.

Multi-Agent Structured Role-Playing Protocol

Core Idea: Decompose code generation into three complementary agent roles—User Agent, Technical Agent, and Quality Assurance Agent—each with distinct responsibilities to enable structured multi-turn deliberation that improves reasoning.

Problem It Solves: SLMs lack the reasoning depth needed for complex logical requirements in code generation. Monolithic inference produces weak outputs. By decomposing the task into specialized roles with explicit turn-taking, the framework creates orthogonal perspectives that collectively mitigate individual model limitations.

How It Works:

  1. User Agent (A_UA): Interprets user intent, clarifies requirements, and proposes initial solution candidates based on the problem specification
  2. Technical Agent (A_TA): Evaluates technical feasibility, suggests optimizations, identifies edge cases, and refines solutions from a correctness perspective
  3. Quality Assurance Agent (A_QA): Reviews proposed solutions against acceptance criteria, identifies bugs, generates test cases, and ensures robustness
  4. Multi-turn deliberation: agents iterate through pre-generation debate phase where each agent provides input before code is generated
  5. Output consensus: final code solution integrates feedback from all three perspectives

Architecture:

flowchart LR
    A[Problem Specification] --> B[A_UA: Propose]
    B --> C[A_TA: Evaluate & Optimize]
    C --> D[A_QA: Review & Test]
    D --> E{Consensus Reached?}
    E -->|No| C
    E -->|Yes| F[Final Code]

Where It’s Useful:

  • Code generation tasks where logical correctness and edge case handling are critical
  • Systems requiring diverse viewpoints to improve reasoning (similar to ensemble approaches)
  • Resource-constrained environments where collaborative inference can boost SLM capability
  • Complex problem-solving domains where decomposition reduces individual model burden

Limitations / Caveats:

  • Requires coordination overhead across three agents; total latency may increase vs. single-agent baseline
  • Agent roles and responsibilities are task-specific; generalization to other domains unclear
  • No clear mechanism for resolving inter-agent disagreements beyond iteration
  • Quality depends heavily on prompt engineering and role definition for each agent

Internal Use Cases:

  • Ranking: ensemble rankers using diverse feature perspectives (A_UA = relevance, A_TA = quality, A_QA = safety)
  • Recommendation: multi-stage filtering with specialized agents (candidate generation, feature ranking, business rule enforcement)
  • Search refinement: debate-based query reformulation using specialized semantic and syntactic agents

Reviewer-Guided Analytical Debugging Loop

Core Idea: Apply post-generation refinement through an iterative debugging cycle where a QA reviewer (A_QA) identifies failures and guides the technical agent in targeted analytical fixes, creating orthogonal pre- and post-generation reasoning.

Problem It Solves: SLMs often produce code with subtle bugs that survive initial generation. The debugging loop provides systematic failure analysis and localized correction, avoiding expensive full regeneration while maintaining code quality.

How It Works:

  1. QA Agent (A_QA) executes generated code on test cases and identifies failures (e.g., assertion errors, logic bugs)
  2. A_QA performs analytical debugging: traces failure root cause, pinpoints buggy code section, generates diagnostic information
  3. Provide failure context (error traces, failing test cases, expected vs. actual output) to Technical Agent (A_TA)
  4. A_TA performs targeted refinement: modifies only the problematic code section(s) guided by diagnostic information
  5. Iterate: re-test refined code; if new failures emerge, repeat steps 2-4; stop when all tests pass
  6. Threshold: if confidence score (from A_QA) falls below 95%, trigger this loop; otherwise skip

Architecture:

flowchart LR
    A[Generated Code] --> B[A_QA: Test & Analyze]
    B --> C{Tests Pass?}
    C -->|Yes| D[Accept Code]
    C -->|No| E[Extract Failure Info]
    E --> F[A_TA: Targeted Fix]
    F --> A
    D --> G[Output]

Where It’s Useful:

  • Code generation with test-driven validation (e.g., HumanEval, MBPP benchmarks)
  • Scenarios where full regeneration is expensive; incremental debugging is preferable
  • Systems requiring interpretable error correction (failure traces guide next steps)
  • Iterative refinement on constrained models where per-iteration cost matters

Limitations / Caveats:

  • Assumes meaningful test cases or error traces are available to guide debugging; weak test coverage limits effectiveness
  • Iterative nature increases latency; worst-case loop may require many rounds
  • Risk of local optima: targeted fixes may not address systemic logic errors requiring broader refactoring
  • Relies on A_TA’s ability to understand and fix issues from diagnostic information; LLM reasoning failures can perpetuate bugs
  • No hard stopping criterion beyond test passage; may produce technically correct but suboptimal solutions

Internal Use Cases:

  • Ranking refinement: iterative feature importance adjustment based on ranking metric failures
  • RecSys: diagnostic loops for improving click-through or conversion metrics by targeted item set modifications
  • Information retrieval: iterative query rewriting based on relevance assessment and failure analysis
