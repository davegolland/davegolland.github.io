FBS: Modeling Native Parallel Reading inside a Transformer

Source: arXiv:cs.CL Published: 2026-01-29 Rank: #3 (score: 1.00, max_sim: 0.82) Reading time: ~15 min Tags: Transformer, LLM, Block-Sparse, Early-Exit, NSA

Abstract

Large language models (LLMs) excel across many tasks, yet inference is still dominated by strictly token-by-token autoregression. Existing acceleration methods largely patch this pipeline and miss core human-reading ingredients: content-adaptive foresight, chunk-structure-aware compute allocation, and train—test consistency for preview/skimming. We propose the \textbf{Fovea-Block-Skip Transformer} (FBS), which injects a causal, trainable loop into Transformers via Parafovea-Attention Window (PAW), Chunk-Head (CH), and Skip-Gate (SG). Across diverse benchmarks, FBS improves the quality-efficiency trade-off without increasing parameters, and ablations show the three modules are complementary.

Chunk-Head (CH)

Core Idea: Groups tokens into fixed-size chunks and applies separate attention heads within each chunk, enabling parallel computation and reducing redundancy by allocating compute proportionally to chunk structure rather than uniformly across all tokens.

Problem It Solves: Standard multi-head attention processes all tokens independently with identical compute. This wastes resources on tokens that could be grouped (punctuation, conjunctions) and misses opportunities to leverage structure. CH allocates compute more efficiently by processing chunks as units, allowing parallel processing and head specialization per chunk.

How It Works:

  1. Partition input sequence into non-overlapping chunks of size c: [tokens 0 to c-1], [tokens c to 2c-1], etc.
  2. For each chunk, apply independent attention computation within that chunk’s tokens
  3. Create “chunk-level” representations by pooling or selecting representative tokens from each chunk (e.g., mean, max, or last token)
  4. Optionally compute limited inter-chunk attention using chunk-level representations before expanding back to token-level

Architecture:

flowchart TD
    A["Input sequence"] --> B["Partition into chunks of size c"]
    B --> C["Intra-chunk attention<br/>in parallel for each chunk"]
    C --> D["Create chunk representations<br/>Pool or select tokens"]
    D --> E["Optional: Inter-chunk<br/>attention on summaries"]
    E --> F["Expand back to token-level"]
    F --> G["Output"]

Where It’s Useful:

  • Documents with clear structural boundaries (chapters, paragraphs, sentences)
  • Batch processing with variable-length sequences (chunks reduce padding overhead)
  • Hierarchical language understanding where local coherence is primary concern
  • Distributed or accelerator-friendly implementations (chunks map to parallel processors)

Limitations / Caveats:

  • Hard chunk boundaries may split semantically meaningful units (e.g., multi-line dialogue)
  • Intra-chunk-only attention means tokens at opposite ends of sequence have no direct attention path
  • Chunk size is fixed hyperparameter; may be suboptimal for variable-structure text
  • Information loss from chunk pooling; fine-grained token distinctions within a chunk are compressed

Internal Use Cases:

  • Candidate ranking where documents are chunked and top chunks are retrieved before full re-ranking
  • Hierarchical recommendation where user history is chunked by time period or session, with attention within/across temporal chunks
  • Reducing memory footprint in retrieval-augmented generation (RAG) by chunking documents for embedding

Skip-Gate (SG)

Core Idea: A learnable gating mechanism that selectively skips full transformer computation for certain positions (tokens or chunks), routing important positions through full attention while using lightweight approximations (e.g., identity or projection) for less important ones.

Problem It Solves: Not all tokens require equal computational investment; many are redundant or context-filler (filler words, repetitions, obvious continuations). SG achieves content-adaptive computation by learning which positions can be processed more cheaply, reducing FLOPs without uniform token-by-token trade-offs.

How It Works:

  1. For each position (token or chunk), compute a gate score g_i ∈ [0, 1] from previous layer hidden state h_i: g_i = σ(W_gate * h_i + b_gate)
  2. If g_i > threshold τ, process position through full transformer (attention, MLP)
  3. If g_i ≤ τ, skip full computation and apply lightweight fallback: h_i^out = identity(h_i) or α * h_i (residual scaling)
  4. Gate parameters (W_gate, b_gate, threshold τ) are learned jointly with the transformer during training

Architecture:

flowchart TD
    A["Hidden state h_i"] --> B["Compute gate score: σ(W_gate * h_i)"]
    B --> C{"g_i > threshold?"}
    C -->|Yes| D["Full transformer block<br/>attention + MLP"]
    C -->|No| E["Lightweight fallback<br/>identity or projection"]
    D --> F["Output h_i^out"]
    E --> F

Where It’s Useful:

  • Inference acceleration on latency-critical systems (skip expensive attention for ~40-60% of positions)
  • Adaptive computation models for heterogeneous workloads (important documents get full processing, summaries get skipped)
  • Token-level importance prediction with learned gates (interpretable via gate activation patterns)
  • Variable-cost inference where budget per sequence is fixed (gates automatically balance trade-offs)

Limitations / Caveats:

  • Gate decisions are based only on local hidden state; global importance signals may be missed
  • Skipped positions may still be semantically important but appear unimportant locally (requires sufficient context in h_i)
  • Training may be unstable if gate threshold and network capacity are not balanced; oscillation between always-skip and never-skip
  • Hard thresholding is non-differentiable; likely uses straight-through estimators or soft gating with learned threshold
  • Risk of degrading quality if too many critical tokens are skipped; requires careful tuning

Internal Use Cases:

  • RecommendationSystems ranking where common user profiles or baseline candidates can be processed with cached/lightweight representations
  • Query-document matching where uninformative query terms (stopwords, low-TF-IDF) skip expensive interaction computations
  • Multi-stage retrieval pipelines where light ranker uses SG to skip expensive candidate evaluation until final stage

These three modules work together synergistically: PAW reduces attention complexity per position, CH groups computation by structure, and SG adaptively skips entire positions—together achieving “native parallel reading” that mimics how humans efficiently process long texts through selective, structured attention.
