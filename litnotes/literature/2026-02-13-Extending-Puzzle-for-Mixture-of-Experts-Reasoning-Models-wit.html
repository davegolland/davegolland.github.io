Extending Puzzle for Mixture-of-Experts Reasoning Models with Application to GPT-OSS Acceleration

Source: arXiv:cs.LG Published: 2026-02-12 Rank: #1 (score: 0.99, max_sim: 0.83) Reading time: ~20 min Tags: Search, RL, LLM, RAG

Abstract

Reasoning-focused LLMs improve answer quality by generating longer reasoning traces, but the additional tokens dramatically increase serving cost, motivating inference optimization. We extend and apply Puzzle, a post-training neural architecture search (NAS) framework, to gpt-oss-120B to produce gpt-oss-puzzle-88B, a deployment-optimized derivative. Our approach combines heterogeneous MoE expert pruning, selective replacement of full-context attention with window attention, FP8 KV-cache quantization with calibrated scales, and post-training reinforcement learning to recover accuracy, while maintaining low generation length. In terms of per-token speeds, on an 8XH100 node we achieve 1.63X and 1.22X throughput speedups in long-context and short-context settings, respectively. gpt-oss-puzzle-88B also delivers throughput speedups of 2.82X on a single NVIDIA H100 GPU. However, because token counts can change with reasoning effort and model variants, per-token throughput (tok/s) and latency (ms/token) do not necessarily lead to end-to-end speedups: a 2X throughput gain is erased if traces grow 2X. Conversely, throughput gains can be spent on more reasoning tokens to improve accuracy; we therefore advocate request-level efficiency metrics that normalize throughput by tokens generated and trace an accuracy—speed frontier across reasoning efforts. We show that gpt-oss-puzzle-88B improves over gpt-oss-120B along the entire frontier, delivering up to 1.29X higher request-level efficiency. Across various benchmarks, gpt-oss-puzzle-88B matches or slightly exceeds the parent on suite-average accuracy across reasoning efforts, with retention ranging from 100.8% (high) to 108.2% (low), showing that post-training architecture search can substantially reduce inference costs without sacrificing quality.

Key Techniques

  • MoE expert pruning
  • window attention replacement
  • FP8 KV-cache quantization
  • post-training reinforcement learning

Puzzle-Extended Neural Architecture Search for Reasoning Model Optimization

Core Idea: Apply post-training neural architecture search to reasoning-focused LLMs by combining heterogeneous MoE expert pruning, selective window attention replacement, FP8 KV-cache quantization, and reinforcement learning recovery to reduce inference cost while maintaining accuracy across varying reasoning efforts.

Problem It Solves: Reasoning models generate long traces that increase serving cost dramatically; traditional per-token throughput metrics are misleading because reasoning token counts vary with model architecture changes, making it unclear whether optimizations actually improve end-to-end efficiency.

How It Works:

  1. Heterogeneous MoE Expert Pruning: Selectively remove experts from mixture-of-experts layers in gpt-oss-120B, reducing active parameters from 120B to 88B while preserving critical reasoning pathways
  2. Selective Window Attention: Replace full-context attention with window attention in non-critical layers to reduce KV-cache memory and computation (context-dependent architectural modification)
  3. FP8 KV-Cache Quantization: Apply calibrated FP8 quantization to key-value caches to reduce memory bandwidth and storage requirements
  4. Post-Training RL Recovery: Run reinforcement learning fine-tuning on the pruned/quantized model to restore accuracy lost during architecture modifications, while constraining generation length to prevent trace explosion

Where It’s Useful:

  • Deploying reasoning-focused LLMs (o1-style models) in production where long traces dominate cost
  • Multi-tenant serving environments where throughput per GPU needs optimization
  • Applications requiring accuracy-speed tradeoffs across different reasoning budgets (e.g., low-effort fast responses vs. high-effort complex reasoning)
  • Single-GPU deployment scenarios where memory and compute are severely constrained

Limitations / Caveats:

  • Requires access to base model architecture and training infrastructure for post-training RL recovery phase
  • Accuracy retention varies by reasoning effort level (100.8%-108.2%), meaning some configurations may show slight degradation
  • Per-token speedups (1.22X-2.82X) don’t guarantee end-to-end request speedups if reasoning traces grow longer
  • Window attention replacement may harm performance on tasks requiring very long-range dependencies
  • Requires careful calibration of FP8 quantization scales to avoid accuracy collapse

Internal Use Cases:

  • LLM Serving Optimization: Apply heterogeneous pruning to reduce serving costs for reasoning-augmented ranking models or recommendation explanation systems
  • Request-Level Efficiency Metrics: Adopt the paper’s accuracy-speed frontier methodology to evaluate RecSys model variants where computation varies by request complexity (e.g., cold-start vs. warm users)
  • Selective Attention Optimization: Use window attention in lower layers of transformer-based rankers where full context may not be necessary
  • Post-Training Recovery: Apply RL fine-tuning after model compression/quantization to recover ranking metrics (NDCG, MRR) lost during efficiency optimizations
