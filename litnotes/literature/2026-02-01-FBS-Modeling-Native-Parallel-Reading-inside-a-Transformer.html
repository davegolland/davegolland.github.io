FBS: Modeling Native Parallel Reading inside a Transformer

Source: arXiv:cs.CL Published: 2026-01-29 Rank: #4 (score: 1.00, max_sim: 0.82) Reading time: ~15 min Tags: LLM, Transformer

Abstract

Large language models (LLMs) excel across many tasks, yet inference is still dominated by strictly token-by-token autoregression. Existing acceleration methods largely patch this pipeline and miss core human-reading ingredients: content-adaptive foresight, chunk-structure-aware compute allocation, and train—test consistency for preview/skimming. We propose the \textbf{Fovea-Block-Skip Transformer} (FBS), which injects a causal, trainable loop into Transformers via Parafovea-Attention Window (PAW), Chunk-Head (CH), and Skip-Gate (SG). Across diverse benchmarks, FBS improves the quality-efficiency trade-off without increasing parameters, and ablations show the three modules are complementary.

Chunk-Head (CH)

Core Idea: A specialized attention head that operates on chunk-level representations rather than individual tokens, allocating compute proportionally to chunk importance rather than treating all tokens equally.

Problem It Solves: Token-level attention allocates compute uniformly across all tokens, but human reading chunks information into meaningful units (sentences, paragraphs). CH addresses this by explicitly modeling chunks and focusing computation on important chunks, enabling chunk-structure-aware compute allocation.

How It Works:

  1. Group input tokens into chunks (fixed-size or hierarchical segments, e.g., by sentence/paragraph boundaries)
  2. Compute a chunk-level importance score (via learned projection or aggregation of token representations within chunks)
  3. Use chunk importance to modulate attention weights: tokens in high-importance chunks receive higher attention allocation
  4. This creates a two-level hierarchy: which chunks matter (coarse), and within chunks, how to attend (fine)

Architecture (if applicable):

flowchart LR
    A["Token Sequence"] --> B["Group into<br/>Chunks"]
    B --> C["Compute Chunk<br/>Importance Scores"]
    C --> D["Modulate Token-Level<br/>Attention by Chunk Score"]
    D --> E["Output"]

Where It’s Useful:

  • Processing long documents where structure (sentences, sections) is meaningful
  • Hierarchical sequence models (e.g., document + paragraph + sentence levels)
  • Information retrieval where document structure indicates relevance

Limitations / Caveats:

  • Requires predefined or learned chunk boundaries; may not align with semantic structure
  • Chunk importance may not perfectly correlate with token-level importance
  • Could bottleneck if chunk computation becomes heavy

Internal Use Cases:

  • Multi-level ranking where documents contain passages, which contain snippets
  • Hierarchical recommendation systems that operate at user-segment, item-category, and item levels
  • Feed ranking with story-level vs. post-level importance allocation

Skip-Gate (SG)

Core Idea: A trainable gating mechanism that decides whether to skip processing (or skip-attention) for certain tokens or chunks, enabling dynamic selection of which computations are necessary based on input content.

Problem It Solves: Transformers perform all computations for all tokens regardless of necessity. Skip-Gate allows selective skipping of compute for less important tokens, improving efficiency while maintaining train-test consistency (the same tokens are skipped during training and inference, avoiding train-test mismatch).

How It Works:

  1. For each token (or chunk), compute a skip score using a learned gating function (e.g., a small neural network on hidden states)
  2. The skip score determines whether to skip processing: if skip_score > threshold, bypass expensive operations (attention, FFN) for this token
  3. Tokens marked as skipped still receive outputs from previous layers but do not contribute to downstream computation
  4. The gate is differentiable (using sigmoid or other smooth gates) so gradients flow during training

Architecture (if applicable):

flowchart LR
    A["Token Hidden State"] --> B["Compute Skip Gate<br/>Score"]
    B --> C{Skip?}
    C -->|No| D["Run Attention<br/>& FFN"]
    C -->|Yes| E["Skip to<br/>Next Layer"]
    D --> F["Output"]
    E --> F

Where It’s Useful:

  • Token-level pruning in long sequence models
  • Early exit strategies where some tokens/sequences finish inference early
  • Efficiency-critical applications (mobile inference, streaming)

Limitations / Caveats:

  • Skip decisions are irreversible; skipped tokens lose processing capacity
  • Requires careful threshold tuning to balance efficiency and accuracy
  • May hurt interpretability (which tokens were actually “read”?)
  • Train-test consistency requires careful handling of randomness during training

Internal Use Cases:

  • Candidate pruning in ranking: mark obviously irrelevant candidates for skipping
  • Lazy evaluation in recommendation engines where some user features don’t need full computation
  • Feed ranking with early-exit for clearly non-relevant items (e.g., already-interacted content)
