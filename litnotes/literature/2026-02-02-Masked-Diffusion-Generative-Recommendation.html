Masked Diffusion Generative Recommendation

Masked Diffusion Generative Recommendation

Source: arXiv:cs.IR Published: 2026-01-27 Rank: #6 (score: 0.92, max_sim: 0.87) Reading time: ~0 min Tags: Uncategorized Code: https://arxiv.org/pdf/2601.19501v2

Abstract

Generative recommendation (GR) typically first quantizes continuous item embeddings into multi-level semantic IDs (SIDs), and then generates the next item via autoregressive decoding. Although existing methods are already competitive in terms of recommendation performance, directly inheriting the autoregressive decoding paradigm from language models still suffers from three key limitations: (1) autoregressive decoding struggles to jointly capture global dependencies among the multi-dimensional features associated with different positions of SID; (2) using a unified, fixed decoding path for the same item implicitly assumes that all users attend to item attributes in the same order; (3) autoregressive decoding is inefficient at inference time and struggles to meet real-time requirements. To tackle these challenges, we propose MDGR, a Masked Diffusion Generative Recommendation framework that reshapes the GR pipeline from three perspectives: codebook, training, and inference. (1) We adopt a parallel codebook to provide a structural foundation for diffusion-based GR. (2) During training, we adaptively construct masking supervision signals along both the temporal and sample dimensions. (3) During inference, we develop a warm-up-based two-stage parallel decoding strategy for efficient generation of SIDs. Extensive experiments on multiple public and industrial-scale datasets show that MDGR outperforms ten state-of-the-art baselines by up to 10.78%. Furthermore, by deploying MDGR on a large-scale online advertising platform, we achieve a 1.20% increase in revenue, demonstrating its practical value.

Adaptive Masking Supervision for Diffusion Training

Core Idea: Dynamically construct masking supervision signals along both temporal and sample dimensions during training to guide the diffusion model in learning multi-dimensional code generation without relying on a fixed autoregressive order.

Problem It Solves: Autoregressive training assumes all users attend to item attributes in the same order, which is unrealistic. Adaptive masking allows the model to learn multiple valid generation paths and attend to different subsets of dimensions depending on context.

How It Works:

  1. Temporal dimension masking: During each training step, randomly select a subset of diffusion timesteps t to train on, rather than deterministically unmasking codes from t=T down to t=0. This exposes the model to varied noising schedules.
  2. Sample dimension masking: For each training example, randomly mask out different subsets of the K code dimensions. Train the model to predict unmasked dimensions conditioned on the masked dimensions as context.
  3. Supervision construction: For a given diffusion step with timestep t, the target is to predict x₀ (clean code vector) or the noise ε added at timestep t, only for the unmasked dimensions
  4. Loss computation: Compute diffusion loss only on unmasked dimensions: L = E_t,mask[||ε_θ(x_t[unmask], t, c_user[unmask]) - ε[unmask]||²] where mask selects which dimensions to supervise

Where It’s Useful:

  • Recommendation with heterogeneous user preferences for attribute importance
  • Scenarios where different user segments prioritize different item features
  • Flexible generation where multiple attribute orderings are equally valid

Limitations / Caveats:

  • Requires careful tuning of masking probability to balance coverage and learning signal
  • May slow convergence if masking is too aggressive (too many dimensions masked)
  • Assumes that masking patterns in training reflect real-world preference patterns

Internal Use Cases:

  • E-commerce recommendations where some users prioritize price while others prioritize ratings
  • Two-stage ranking where first-stage and second-stage rankers may attend to different features

Warm-up-Based Two-Stage Parallel Decoding Strategy

Core Idea: Decompose inference into a warm-up stage (using fewer diffusion steps to generate coarse code estimates) and a refinement stage (parallel refinement of all K dimensions) to enable efficient real-time recommendation generation.

Problem It Solves: Autoregressive decoding is sequential and slow at inference time, violating real-time latency budgets in production systems. Diffusion models typically require many forward passes. This strategy reduces inference latency by parallelizing dimension generation across stages.

How It Works:

  1. Stage 1 - Warm-up (coarse generation):

    • Run the diffusion model for N_warm << N_total diffusion steps (e.g., N_warm=5)
    • Start from pure noise x_T = (noise, noise, …, noise) across all K dimensions
    • Generate a coarse estimate of all K code dimensions in parallel
    • Output: partial code estimates ĉ_warm = [ĉ¹_warm, ĉ²_warm, …, ĉ^K_warm]

  2. Stage 2 - Refinement (parallel refinement):

    • Use the warm-up codes as initialization: x_{N_warm} = ĉ_warm + noise_schedule(N_warm)
    • Run (N_total - N_warm) additional diffusion steps in parallel across all K dimensions
    • Each dimension is refined independently using its corresponding learned score network
    • No sequential dependency between dimensions

  3. Final code selection: Argmax over vocabulary for each dimension: final_code^j = argmax_v p(c^j | context, noise_level)

Architecture:

flowchart LR
    Input["User context<br/>(embedding)"] --> WarmUp["Stage 1: Warm-up<br/>(N_warm diffusion steps)<br/>Parallel across K dims"]
    WarmUp --> CoarseEst["Coarse Code Estimates<br/>[ĉ¹, ĉ², ..., ĉᴷ]"]
    CoarseEst --> Refine["Stage 2: Refinement<br/>(N_total-N_warm steps)<br/>Parallel dimension refinement"]
    Refine --> FinalCodes["Final Codes<br/>[c¹, c², ..., cᴷ]"]
    FinalCodes --> Output["Recommended Item"]

Where It’s Useful:

  • Real-time recommendation systems with strict latency SLAs (e.g., online advertising, search results)
  • Large-scale production systems where inference speed directly impacts revenue
  • Applications requiring per-request personalization within millisecond budgets

Limitations / Caveats:

  • Two-stage pipeline adds implementation complexity compared to single-stage generation
  • Warm-up stage quality directly impacts refinement stage; poor initialization may hurt final quality
  • Requires empirical tuning of the warm-up step count N_warm to balance quality vs. latency
  • May require retraining if warm-up ratio changes significantly

Internal Use Cases:

  • Sponsored search recommendations: first stage generates candidate items in ~2ms, second stage refines for ~3ms total latency
  • Feed ranking: warm-up stage filters to top-K items, refinement stage personalizes rankings within latency budgets
